# Phase 1 plan — extraction

Implementation plan for phase 1 of [the osr-forge spec](spec.md). Phase 1 delivers the two model-calling extraction stages: survey (stage 1), one structured-output pass over the whole module producing the index that plans everything downstream, and content (stage 2), per-level batched extraction of keyed areas — description, encounters, traps, treasure, features, connections — each with source pages and self-assessed confidence. The roadmap milestone: one full eval module — JN1 The Chaotic Caves, 48 pages, the license-verified phase 0 spike module — extracting with credible per-area output and source pages. Every chunking, schema, and prompt decision below rests on the measured behavior in [docs/foundry-capabilities.md](foundry-capabilities.md) and on the quality hazards the spike's extraction fixtures exposed; those hazards are woven into the work items that defuse them so the implementer doesn't rediscover any of them.

## Scope

In scope:

- Stage-cache contract models (`contracts/stages.py`) for `stages/survey.json` and `stages/areas.<dungeon>.<level>.json`
- Plumbing: `Workdir` stage-path helpers, run-tracking helpers (`RunMeta.with_stage`/`with_model`, `track_stage`), new `ConversionSettings` knobs, `ExtractionError`
- `pages.py` — the shared interleaved text-plus-image request-part builder
- `survey.py` — survey schema, prompt rules, request builder, normalization to canonical ids/keys, the stage function
- `content.py` — batch planner, per-batch schemas, request builder, missing-key follow-up, per-level cache writes, the stage function
- `tools/extract/` runner for live extraction runs; recorded minimod fixtures with committed page renders and golden caches
- Replay-grade chaotic-caves fixtures for the real phase 1 prompts over the already-committed 8 pages
- The milestone and verification runs: a full live 48-page JN1 extraction with its evidence committed inside the fenced asset directory and gated by a zero-network credibility-floor test, plus live verification runs over two additional licensed, non-redistributable modules (nothing committed)
- Tests for all of the above, green in CI with no network

Out of scope (later phases): monster resolution, geometry synthesis, assembly, report *production*, SVG previews, and `convert()` plus the `osrforge` console script (phase 2 — the phase 0 plan's "console script arrives with the first end-to-end `convert`, phase 1 at the earliest" resolves to phase 2, because end-to-end `convert` requires phase 2's assemble; work item 6's repo-only runner covers manual driving until then); overrides *application*, `rerun`/resume, playability lint, `estimate` (phase 3); survey chunking past the single-request guard and the eval corpus/scoring harness (phase 4); structured treasure and trap parsing into osrlib specs (phase 2, with the `treasure_unparsed` flag as the standing escape).

Spec impacts: none — no spec edits are needed. One pinned reading is recorded here instead, mirroring phase 0's geometry-stage reading: work item 1 reads the workdir layout's "cached raw stage outputs — the LLM's actual answers" as the extraction stage's validated-and-normalized output, with the model's original spellings preserved in the cache. Phase 2 confirms the reading when assembly consumes the caches, or drives a spec fix.

## Work items

### 1. Stage-cache contracts — `contracts/stages.py`

Stage caches are cross-phase wire formats — content reads survey's cache; phase 2's monsters and assemble stages read both — so their models live in `contracts/` (the established home for anything serialized between phases), in one module that phase 2's monsters-cache model later joins. No stage module ever imports another stage module. All models are frozen pydantic v2 with `extra="forbid"`, written via the pinned `write_json_artifact`.

- **Pinned reading — normalized, not wire-raw.** The spec's workdir comment calls `stages/` "cached raw stage outputs — the LLM's actual answers". Read as: the extraction *stage's* output, frozen before any downstream (human or deterministic) processing — and normalization (work item 4) is part of the extraction stage, which `contracts/report.py` already pins ("phase 1's extraction normalization enforces it at the source"). The reading is forced by mechanics: the content stage's cache *filenames* (`areas.<dungeon>.<level>.json`) and its per-batch key enums need canonical dungeon ids and area keys before content runs, and overrides address areas by canonical `<dungeon>/<level>/<key>`. Nothing is lost: every renamed identifier keeps the model's original spelling in a `source_label` field, and a host that wants the literal wire answers wraps its provider in `RecordingProvider` — that is that class's published role.
- `stages/survey.json` = `SurveyIndex`: `schema_version`, `title`, `hooks: tuple[str, ...]`, `town: TownInfo` (`name` — empty allowed, see work item 4 — and `description`), `dungeons: tuple[SurveyDungeon, ...]`, `monster_names: tuple[str, ...]`.
    - `SurveyDungeon`: `id` (canonical slug), `name` (the model's printed name, untouched), `levels: tuple[SurveyLevel, ...]` (min length 1). No `source_label` here: the id is derived from `name`, which is preserved on the same model, so a label field could never carry information `name` doesn't — dead accommodation.
    - `SurveyLevel`: `number` (int, ≥ 1, unique per dungeon), `map_pages: tuple[int, ...]` (pages showing this level's map — load-bearing for content's direction extraction, work item 5), `areas: tuple[SurveyArea, ...]`.
    - `SurveyArea`: `key` (canonical, unique per level), `name`, `source_label: str | None`, `kind: Literal["room", "corridor", "cave", "landmark", "other"]`, `source_pages: tuple[int, ...]`.
- `stages/areas.<dungeon>.<level>.json` = `LevelContent`: `schema_version`, `dungeon_id`, `level_number`, `areas: tuple[AreaContent, ...]`.
    - `AreaContent`: `key`, `description`, `encounters: tuple[AreaEncounter, ...]`, `trap: str | None`, `treasure: tuple[str, ...]`, `features: tuple[str, ...]`, `connections: tuple[AreaConnection, ...]`, `source_pages: tuple[int, ...]`, `confidence` (float in [0, 1]).
    - `AreaEncounter`: `monster: str` (free text — name resolution is phase 2's monsters stage), `count_fixed: int | None` (≥ 1), `count_dice: str | None`, `count_note: str | None`. The three count fields are independent optionals; the cache stores what the model said. Phase 2's monster stage owns the mapping onto osrlib's exactly-one-of rule (prefer dice when both are set; flag when neither is), and demotes any `count_dice` osrlib's dice parser still rejects to `count_note` plus a flag — defense in depth behind the schema pattern below.
    - `AreaConnection`: `to_key: str` (free string — connections may cross batches or levels; the prompt instructs canonical keys from the survey excerpt, and dangling references are assembly's job, surfacing as `connection_ambiguous`), `direction: Literal["north", "south", "east", "west", "up", "down", "unknown"]`.
- **Versioning pinned: caches carry `schema_version` only, not `osrforge_version`.** The producing package version is already recorded once per workdir in `run.json`; duplicating it per cache would break golden-file byte-equality (work item 6) on every package version bump without adding information. The spec pins the version *pair* for `report.json` and `run.json` only and is silent on caches, so this is an in-bounds choice, not a deviation.
- **Usage and model identity live only in `run.json`** (`StageStatus.usage` per stage; `RunMeta.provider`/`model_id` at the top). Usage varies with provider retries, so putting it in caches would let byte-identical extractions produce differing caches — poison for the byte-stability property assembly purity rests on. Caches carry facts; assembly (phase 2) derives report flags from them deterministically (for example `direction: "unknown"` → `connection_ambiguous`).

### 2. Plumbing — workdir, run tracking, settings, errors

- `Workdir` gains the missing stage-cache path helpers (no other module builds these paths by hand): property `survey_json` → `stages/survey.json`; method `areas_json(dungeon_id: str, level_number: int)` → `stages/areas.<dungeon>.<level>.json`. Filename unambiguity is guaranteed by the slug alphabet — no dots or slashes in ids (work item 4). Stage functions create `stages/` on first write.
- `contracts/run.py` gains two pure helpers on the frozen `RunMeta` — `with_stage(stage: Stage, status: StageStatus) -> RunMeta` and `with_model(provider: str, model_id: str) -> RunMeta` — the `model_copy(update=...)` pattern; still contract territory because they're pure transformations on a contract model.
- `workdir.py` gains `track_stage(workdir, stage)`, a context manager owning the running → completed/failed choreography (its charter already includes `run.json` I/O). On enter it writes the stage `running` with `started_at`; it yields a small mutable `StageTracker` with `add_usage(usage: TokenUsage)` and `set_model(provider: str, model_id: str)`; on clean exit it writes `completed` with `finished_at` plus the accumulated usage and applies the provider/model identity; on exception it writes `failed` with `error=str(exc)` and re-raises, leaving upstream artifacts untouched. Timestamps are legal here and only here — `run.json` is operational metadata, per phase 0's pin.
    - Provider identity pinned: `RunMeta.provider = type(provider).__name__`. The `ModelProvider` protocol stays exactly as narrow as the spec's — no name attribute added — and a `FixtureProvider` run honestly records `"FixtureProvider"`. A recording run (work item 6's runner wraps `FoundryProvider` in `RecordingProvider`) records `"RecordingProvider"` — accepted and noted rather than special-cased: unwrapping would couple stage code to a concrete provider class, and `model_id` still records the real deployment. `model_id` is the most recent response's `model_id` (last-write-wins if a deployment updates mid-run — acceptable, noted).
    - Usage accumulation trap, defused: `FoundryProvider` already folds schema-retry attempts into each response's usage, so trackers sum once per response — summing anywhere else double-counts.
- `settings.py`: `ConversionSettings` gains `content_batch_pages: int = 8` (≥ 1; the spec's "content-pass batch size" knob — the spike's 8-page batches ≈ 16K input produced clean per-area output) and `survey_max_pages: int = 150` (≥ 1; the single-request survey guard from `docs/foundry-capabilities.md` — beyond ~150 pages the request crosses the 272K-token pricing cliff). Fuzzy-match threshold and top-k stay deferred to phase 2, their consumer. Both knobs have defaults, so `run.json` settings echoes written by phase 0 still parse.
- Stage functions read settings from `run.json`'s settings echo, not from a parameter — one source of truth per workdir; settings-drift detection between stages stays phase 3's `rerun` job.
- `errors.py` gains `ExtractionError(OsrForgeError)`: runtime failures of the extraction work itself — the source exceeds the survey guard, or the survey yields zero dungeons or zero areas (a dead conversion; osrlib requires at least one dungeon). Provider failures keep propagating as `ProviderError`/`SchemaValidationError`; calling a stage on a workdir whose upstream stage isn't `completed` is programmer misuse and raises stdlib `ValueError`, per phase 0's pinned rule.

### 3. Shared page parts — `pages.py`

One public function: `page_request_parts(workdir: Workdir, page_numbers: Sequence[int]) -> tuple[TextPart | ImagePart, ...]` — for each page, in the given order, a `TextPart` of `"[page N]\n"` plus the page's extracted text, immediately followed by the page's `ImagePart`. Missing page files raise `ValueError` (misuse: preprocess didn't run, or a bad page number).

- **Including the text layer is load-bearing, not a nicety**: the spike's images-only survey request (`probe.survey-schema`, 2 pages, no text parts) produced one dungeon, three areas, and one monster name, while the text-plus-image request over the same module's pages produced the full index. The comparison proves text inclusion matters (the probes also differed in page count); per-page interleaving specifically is the pinned arrangement — untested in isolation but the shape every successful extraction probe used. Both stages and phase 2's passes build parts only through this function.
- The `[page N]` marker is emitted even when the text layer is empty — the scanned-module path, exercised by minimod's raster map page (the one page generated with no text objects). The markers also *define the page-number space*: printed page numbers visible in the images differ from PDF page numbers (chaotic-caves PDF page 38 prints page number 36), so every prompt states that `source_pages`/`map_pages` refer to the `[page N]` markers.
- It lives in its own module: `Workdir`'s charter is layout plus `run.json` I/O, `providers/` is the vendor seam, and both stages need it — a one-function module matches house scale (`versioning.py`).

### 4. Survey stage — `survey.py`

The survey JSON schema, refined from the spike's proven shape: `title`, `hooks` (array of string), `town {name, description}`, `dungeons` (array of `{name, levels: [{number (integer), map_pages (array of integer), areas: [{key, name, source_pages, kind (enum: room, corridor, cave, landmark, other)}]}]}`), `monster_names` (array of string); `additionalProperties: false` throughout, everything required. Two deliberate changes from the spike schema: **no model-supplied dungeon `id`** (canonical ids are osr-forge's construct, derived by slugging `name` — asking the model for an id just created a second thing to normalize), and **`map_pages` added per level** (feeds work item 5's direction extraction). The schema is a fraction of the proven e512-d8 budget (~9 KB canonical, enums ≤ 512, `$ref` nesting ≤ 8); native `json_schema` mode is requested but honored-not-guaranteed, so the adapter's validate-and-retry path stays mandatory — both per `docs/foundry-capabilities.md`.

System-prompt rules, each pinned against an observed failure in the spike fixtures:

- **Town is never a dungeon.** The spike survey emitted `"Town Key"` as a dungeon. Rule: a dungeon is a keyed adventuring site (caves, ruins, lairs); the town/base and its buildings are described only in `town`. The prompt is the lever — normalization does not try to detect town-shaped dungeons; residual mistakes are human-correctable through overrides.
- **One dungeon per independently keyed map — decided: JN1's lairs A–E are five dungeons, not five levels.** Rule: maps connected internally by stairs or shafts are levels of one dungeon; separate lairs with their own maps and entrances are separate dungeons. Rationale: it matches the module's own presentation, needs no merge heuristic, osrlib's `Adventure` takes 1..n dungeons with per-dungeon `travel_turns`, and phase 2's geometry synthesis gets connected per-dungeon graphs (a merged single level would have disconnected components and fail reachability lint). Implementers must not "fix" the spike's lair-splitting — it is the pinned behavior.
- **Hooks spelled out.** The spike returned empty `hooks` and an empty `town.name`. The prompt defines hooks (rumors, jobs, and reasons the party goes — usually in the introduction or background) and permits an empty `town.name` when the town is genuinely unnamed; osrlib's required `Adventure.name`/`TownSpec.name` get defaults-plus-flags at assembly (phase 2). Honest caveat, to verify on the milestone run: the committed 8-page subset excludes the module's introduction, so the empty hooks may be an excerpt artifact rather than a model weakness.
- `source_pages`/`map_pages` refer to the `[page N]` markers (work item 3).

Normalization — public, pure, provider-free functions in `survey.py`, because `contracts/report.py` pins normalization at the source and the source is this stage:

- `canonical_slug(text: str) -> str`: NFKD-decompose, drop non-ASCII, lowercase, collapse every run outside `[a-z0-9]` to a single `-`, strip leading/trailing `-`. Result matches `^[a-z0-9]+(-[a-z0-9]+)*$` or is empty (callers apply fallbacks). The alphabet is the point: no `/` (the `AreaAddress` grammar forbids it), no `.` (so `areas.<dungeon>.<level>.json` filenames parse unambiguously and tags stay within the `ModelRequest` tag charset), and lowercase (so a hand-edited `overrides.yaml` can't alias `4a` against `4A`). From the real fixture: `"A. Orc Lair"` → `a-orc-lair`, `"4a"` → `4a`, `"East, North and South Gates"` → `east-north-and-south-gates`.
- `normalize_survey(raw: dict, page_count: int) -> SurveyIndex`, deterministic: dungeon ids = slug of `name`, empty → `dungeon-<position>` (1-based, document order). Collision handling, pinned as **reserve-then-bump**: first, every slugged form is reserved — each area whose slugged key is the *first* occurrence of that form keeps it; then each later duplicate takes the lowest `-2`, `-3`, … suffix that collides with neither a reserved form nor an already-assigned key. Reserving first means a genuine printed key never gets renamed just because a duplicate elsewhere bumped into it. The naive suffix-by-duplicate-count reading emits duplicates on model keys `"5"`, `"5"`, `"5-2"`; reserve-then-bump yields `5`, `5-3`, `5-2` (the second `"5"` skips `5-2` because area three reserved it). This exact case goes in the normalization tests. Level numbers invalid or non-unique within a dungeon → renumbered 1..n in listed order. Area keys = slug of the model's key, empty → `area-<position>`; within-level collisions bump the same way — the spike's two `key: "5"` areas become `5` and `5-2`. `source_label` preserves the model's original string wherever the canonical form differs, and the human-facing `name` is never touched, so a reviewer can still cross-reference the printed module. Page references outside 1..page_count (models hallucinate them) are dropped; page lists are deduplicated and sorted. Zero dungeons or zero areas after normalization → `ExtractionError`.
- Every emitted `<dungeon>/<level>/<key>` triple must round-trip through `contracts/report.py`'s `AreaAddress` — asserted in tests, which is the point of the whole exercise.

The stage function `survey(workdir: Workdir, provider: ModelProvider) -> SurveyIndex`: require preprocess `completed` (else `ValueError`); enforce the `survey_max_pages` guard (else `ExtractionError`, before any model call); under `track_stage`, build the request over all pages 1..page_count via `build_survey_request`, call the provider, normalize — and only then, **on success**, delete every stale `stages/areas.*.json` (a re-run survey can change canonical ids, orphaning old content caches) and write `stages/survey.json`. Clearing on success rather than on entry means a transient provider failure on a re-run leaves the previous consistent survey-plus-content cache set intact; survey's own write is a single atomic artifact, so nothing needs pre-clearing. (Content clears first instead — work item 5 — because its incremental per-level writes could otherwise leave a stale mixture.) Request tag: `survey`.

`build_survey_request(...) -> ModelRequest` is public and pure, as is work item 5's `build_batch_request`: the tools runner and the fixture tests must build fingerprint-identical requests without duplicating prompt code — the lesson behind phase 0's fixture-digest amendment.

### 5. Content stage — `content.py`

Batch planning, per `(dungeon, level)` in survey order:

- Page set = union of the level's areas' `source_pages` (already clamped by survey normalization), deduplicated and sorted ascending. Batches are **sliding windows over that sorted list** — contiguity of the page numbers themselves is irrelevant; a sparse set like {3, 7, 20, 21} windows exactly like a dense one. The first batch is the first `content_batch_pages` entries; each subsequent batch starts at the previous batch's last entry (stride `content_batch_pages − 1`), so consecutive batches share exactly one page and **every batch has at most `content_batch_pages` pages including the shared one** — the knob is a true request-size cap. The overlap exists so an area whose text spans a batch boundary is fully visible to its owning batch.
- The level's `map_pages` are appended to every batch — after the batch's own pages, ascending, excluding any already in the batch — so the part sequence is batch pages then map pages. The map is how the model answers `direction` when the prose is silent.
- Each area is assigned to exactly one batch — **the earliest batch whose page set contains the area's first source page** (the boundary page sits in two batches; earliest-wins breaks the tie deterministically) — enforced through the batch schema's key enum, so the cross-batch merge is pure concatenation and a duplicate key across batches is a code bug (asserted), not a model behavior. Areas with no source pages are assigned to the level's last batch and named in its prompt. A level whose areas have no source pages but whose `map_pages` is non-empty gets a single batch of just the map pages, carrying all the level's areas. A level with no pages at all (no source pages, no map pages) writes an empty `LevelContent` without a model call; its areas stay content-less for phase 2's placeholder-plus-flag treatment.
- Per-batch JSON schema: `areas` (array) of `{key, description, encounters: [{monster, count_fixed (integer|null), count_dice (string|null), count_note (string|null)}], trap (string|null), treasure (array of string), features (array of string), connections: [{to_key, direction (enum: north, south, east, west, up, down, unknown)}], source_pages, confidence (number, minimum 0, maximum 1)}` — with **`key` constrained by an `enum` of exactly the canonical keys that batch must extract**. The model cannot invent, misspell, or duplicate a key the code didn't ask for; a level's area count is far below the proven 512-value enum budget. Should the model return the same key twice in one response anyway, the first occurrence wins (deterministic). The `count_dice` pattern mirrors osrlib's dice grammar exactly — `^([1-9][0-9]{0,2})?d(2|3|4|6|8|10|12|20|100)([+-](0|[1-9][0-9]{0,5}))?$` (optional 1–999 count, the allowed die sizes, optional canonical modifier) — so a schema-valid dice string is an osrlib-parseable dice string; a looser pattern like `2d7` or `1d6+07` would pass the cache and fail `KeyedMonster` validation at phase 2. Non-conforming counts belong in `count_note`, which the prompt says.
- `trap` and `features` are new versus the spike schema — the spec's content-stage description requires them (osrlib's `AreaSpec` has a single optional `trap` and a `features` tuple, hence one nullable string and an array of short strings; structured `TrapSpec`/`FeatureSpec` mapping is phase 2's). Treasure stays free text — the `treasure_unparsed` flag exists precisely because treasure parsing is imperfect, and structuring it belongs with phase 2's `AreaTreasureSpec` mapping. Counts are structured *now* (the model reading "2d4 orcs" is a far better parser than phase 2 code reading the spike's `"1d6 per hour on 1-2 on 1d6"` — anything unparseable lands in `count_note`), while monster names stay free text for phase 2's monsters stage.

Batch prompt content, pinned against the spike's observed junk:

- A compact survey-index excerpt for the level — one line per area in the batch: canonical key, printed name, `source_label` (the printed key), expected pages — so the model maps printed labels back to the canonical keys the schema demands.
- **Encounter definition**: encounters are the area's hostile or monstrous inhabitants with stat blocks; ordinary townsfolk and named NPCs without a creature type are not encounters — they belong in the description (the spike emitted `"Hunter father"`, `"Widow woman"` as encounters). Use the creature type as printed ("goblin"), putting proper names ("Snagg") in the description. Wandering-monster tables are not keyed encounters.
- **Direction**: derive `direction` from the map when the text is silent; `unknown` only when neither says (the spike, with no map page in context, returned almost all `unknown`).
- `source_pages` refer to the `[page N]` markers; confidence is a self-assessment in [0, 1].

Orchestration: after each batch response, code checks key coverage; keys the model skipped get one follow-up request — same parts as the batch, the prompt naming just the missing areas, and **the schema's key enum containing exactly the missing keys** (so the follow-up cannot re-answer keys the batch already covered); keys still missing after that are absent from the cache — the survey index remains the authority on what exists, and phase 2's assembly emits placeholders flagged `low_confidence` (pinned reading for phase 2, recorded here). Each `areas.<dungeon>.<level>.json` is written as its level completes, so a mid-stage failure keeps finished levels (the spec's "a failure stops there, keeps everything upstream"; stage-level resume is phase 3's `rerun`). Request tags: `content.<dungeon-id>.<level>.b<NN>` (two-digit, 1-based) and `content.<dungeon-id>.<level>.b<NN>.retry` — slugs contain no dots, so tags parse unambiguously and stay within the tag charset.

The stage function `content(workdir: Workdir, provider: ModelProvider) -> tuple[LevelContent, ...]`: require survey `completed` and its cache present (else `ValueError`); under `track_stage`, delete every stale `stages/areas.*.json`, then plan, extract, and write per level. It reads `stages/survey.json` rather than taking the index as a parameter — the cache is the contract, matching `rerun` semantics.

### 6. Extraction runner and recorded fixtures — `tools/extract/`, minimod, chaotic-caves

`tools/extract/run_extraction.py` plus a README: drives `preprocess` → `survey` → `content` over a PDF with `FoundryProvider`, wrapping it in `RecordingProvider` only when a fixtures-directory flag is given — **recording is opt-in**, because work item 7's verification runs over non-redistributable modules must leave no recorded module text behind. Repo-only, never packaged, never in CI — exactly like `tools/spike/`. Recording sessions are manual, named work; the README documents re-runnable commands, because **any prompt or schema edit changes request fingerprints and strands fixtures — re-record, and never rebuild fixture requests from fresh renders** (a pdfium/Pillow bump would strand them; phase 0's fixture-coupling rule).

- **Minimod recording — the pipeline-test centerpiece.** Record real `survey()` + `content()` runs over the committed CC0 `tests/assets/minimod/minimod.pdf` (5 pages, 6 keyed areas, dimensions, monsters, treasure, one text-less map page). Commit: the exact page renders recorded against (`tests/assets/minimod/pages/`, tiny), the fixtures (`tests/assets/minimod/fixtures/`), and golden caches (`tests/assets/minimod/expected/` — `survey.json` plus each `areas.*.json`) that pin normalization output byte-for-byte. Update `tests/assets/README.md`: regenerating `minimod.pdf` or bumping pdfium/Pillow strands these fixtures; re-record via the documented commands.
- **Phase 1 chaotic-caves recordings live in their own directory: `tests/assets/chaotic-caves/fixtures-extract/`.** `fixtures/` remains phase 0's, owned by `test_spike_fixtures.py` — whose evidence/replay split test sweeps *every* fixture in that directory and fails any non-`REPLAY_GRADE_TAGS` fixture that is fully reconstructable from committed pages. Phase 1's excerpt fixtures are fully reconstructable by design, so dropping them into `fixtures/` would turn CI red; the separate directory keeps the phase 0 test untouched. The new phase 1 fixture test owns `fixtures-extract/` and classifies by an explicit tag list — the excerpt fixtures replay; everything else in the directory is milestone evidence with no replay promise, no reconstructability sweep applied.
- **Chaotic-caves replay-grade fixtures for the real phase 1 prompts** over the already-committed 8 pages (0008, 0022–0027, 0038): one survey request and one content batch, both built through the real `build_survey_request`/`build_batch_request`. The chain is deterministic end to end: the test replays the excerpt survey fixture, normalizes its data (with `page_count=48`, the real module's), plans the first content batch, builds the batch request, and replays the content fixture — a mini pipeline over committed pages with zero network. Closure over the committed subset is pinned, not hoped for: the runner's excerpt mode filters every page reference in the normalized index down to the committed subset before planning (a documented step in the shared code path), and both the recorder and the test source page parts exclusively from `tests/assets/chaotic-caves/pages/` — so an in-range model reference to an uncommitted page (say 21 or 39) cannot make the batch request unbuildable at replay time.
- The excerpt survey and batch requests take explicit page lists (the committed pages keep their original 0008/0022/… numbering, so a fabricated contiguous workdir can't represent them); the full stage functions are exercised end to end by minimod, whose entire 5-page workdir is representable from committed assets.

### 7. The milestone and verification runs

The spec milestone — one full eval module extracting with credible per-area output and source pages — is evidenced by machine-checkable committed artifacts, not vibes, and doneness is verified against three modules, not one:

- **JN1 (the committed-evidence run).** One full live 48-page run through the runner (the local, gitignored, sha256-verified PDF), recording into `fixtures-extract/`. Committed into the fenced `tests/assets/chaotic-caves/` (its text and maps are CC BY-SA 4.0; committed content currently 4.4 MiB of the directory's 10 MiB budget — headroom verified): the fixtures of every request, and the produced stage caches under `tests/assets/chaotic-caves/stages/`. These fixtures carry no replay promise — work item 6's explicit tag list classifies them as milestone evidence regardless of which pages happen to be committed (a content batch whose pages all fall inside the committed subset *would* be reconstructable, so classification must not hinge on reconstructability).
- A **credibility-floor test** (zero network) loads the committed caches, validates them against `contracts/stages.py`, and asserts a floor: every survey area's `<dungeon>/<level>/<key>` parses as an `AreaAddress`; every extracted area has a non-empty description, `source_pages` within 1..48, and confidence in [0, 1]; no dungeon id in a small town-shaped denylist (`town`, `town-key`); and module-specific floors finalized against the actual run (on the order of: at least 4 dungeons — JN1 keys five lairs — and at least 40 extracted areas). The caches are static committed files, so these assertions are a regression gate on the evidence, not on the live model.

### 8. Tests

- **contracts/stages**: round-trips (`model_validate(model_dump())`); confidence and count bounds; unknown-key rejection; every model frozen.
- **plumbing**: `Workdir.survey_json`/`areas_json` paths; `track_stage` choreography (running with `started_at` on entry; completed with usage/provider/model identity on exit; failed with the error and re-raise on exception; upstream files untouched); `with_stage`/`with_model` purity.
- **pages**: interleaved order; `[page N]` markers on empty-text pages; `ValueError` on missing pages.
- **normalization tables**: slug rules (case, punctuation, non-ASCII, empty results); collision suffixing including the real two-`"5"` case and the bump-until-free case (`"5"`, `"5"`, `"5-2"` → `5`, `5-3`, `5-2`); label keys (`"East, North and South Gates"`); level renumbering; page clamping/dedup/sort; the `^[a-z0-9]+(-[a-z0-9]+)*$` invariant; `AreaAddress` round-trip of every emitted address; zero-dungeon/zero-area → `ExtractionError`.
- **survey stage (scripted stub provider)**: hazard-shaped model data (duplicate keys, empty and out-of-range `source_pages`, duplicate dungeon names) lands normalized in the cache; `run.json` transitions; downstream-cache clearing; guard violation raises before any provider call; stub failure → stage `failed`, upstream intact, no partial `survey.json`.

- **pipeline replay (minimod)**: the real `survey()` + `content()` against `FixtureProvider` over a workdir fabricated from the committed pages — byte-stable across two runs and byte-equal to the committed goldens.
