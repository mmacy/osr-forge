# Phase 2 plan — a playable draft

Implementation plan for phase 2 of [the osr-forge spec](spec.md). Phase 2 delivers the pipeline's back half: monster resolution (stage 3), deterministic geometry synthesis (stage 4), assembly with `validate_adventure` and report production (stage 5), SVG previews, and the first end-to-end `convert()` with the `osrforge` console script. The roadmap milestone: a converted module loads and plays in osrlib's example TUI crawler. Every mapping and synthesis decision below rests on the osrlib 1.1.0 surfaces verified against the checkout at `~/repos/osrlib-python` and on the committed phase 1 stage caches (JN1's 137 areas, minimod's 6); the osrlib hazards that would silently kill the milestone are woven into the work items that defuse them.

Three osrlib facts shape everything and are worth stating before the work items:

- **A draft with a dangling monster id cannot play at all.** `GameSession.new` calls `validate_adventure(adventure, load_monsters(), load_equipment())` before the session exists (`session.py:273`) and raises `ContentValidationError` on the first call. So "loads and plays" requires a draft that *passes* content validation — which forces the unresolved-encounter decision in work item 5.
- **osrlib grids are walls by default.** An edge absent from `LevelSpec.edges` is a wall, and the boundary is an implicit wall — a bare grid of cells is impassable. Geometry synthesis must emit `open` edges between adjacent cells *within* each area, along every corridor, and at every room–corridor junction, or the party cannot leave the entrance cell.
- **osrlib has no adventure-file loader.** An adventure is a pydantic model built in code; the example TUI constructs its content via `build_adventure()`. Loading `adventure.json` is osr-forge's contract (`check_document` + `Adventure.model_validate`), and the milestone's "loads in the example TUI" means substituting our loaded adventure for the example's built one (work item 8).

## Scope

In scope:

- Stage-cache contract for `stages/monsters.json` (`contracts/stages.py`), plus the additive module-level `flags` field on `ExtractionReport`
- Plumbing: `Workdir` monsters-cache and preview path helpers, upstream-stage clearing of the monsters cache, two new `ConversionSettings` knobs
- `monsters.py` — stage 3: name normalization, the four resolution tiers (exact, alias, fuzzy, LLM), the cache, and the stage function
- `geometry.py` — stage 4: dimension parsing, the connection graph, BFS placement with corridors, edges, entrance, transitions, and the structural postconditions
- `previews.py` — the deterministic SVG level renderer
- `assemble.py` — stage 5: content mapping (encounters and counts, traps, features, treasure), the `Adventure` build, `validate_adventure`, report production, and artifact writing
- `convert.py` and `cli.py` — the end-to-end `convert()` with `on_progress`, the `osrforge` console script (`convert`, `assemble`, `preview`), and the façade exports
- `tools/extract/` extension for the monsters stage and full-chain runs; the JN1 monsters recording and assembled goldens; minimod full-chain goldens
- The milestone: a zero-network playability smoke test in CI plus a manual play session of converted JN1 in osrlib's example TUI crawler, recorded as an amendment
- Tests for all of the above, green in CI with no network

Out of scope (later phases): overrides *application*, `rerun`/resume, playability lint (including the full smoke delve along a computed path), `estimate`, and the `check`/`rerun`/`estimate` CLI subcommands (phase 3); survey chunking and the eval corpus/scoring harness (phase 4); door *extraction* — synthesized edges are all `open` in v1 and doors arrive through geometry overrides (phase 3) or a future vision pass (see the spec impacts below); structured trap *mechanics* (damage, saves, conditions — free-text traps ride `TrapEffect.manual`; mechanical trap specs are override territory); custom `MonsterTemplate` emission (the spec's standing open question).

**Constraint carried through every work item: phase 2 must not touch the phase 1 prompts or schemas.** `SURVEY_SYSTEM`, `SURVEY_SCHEMA`, `CONTENT_SYSTEM`, and `batch_schema` are frozen — any edit changes request fingerprints and strands every committed survey/content fixture (minimod, the chaotic-caves replay chain, and the milestone evidence). Everything phase 2 needs is derivable from the caches those prompts already produce.

Spec impacts, applied in this PR:

- **§ Pipeline, the monsters stage (stage 3):** the sentence "Unresolved names become report flags; `validate_adventure` rejects dangling ids, so publishing anywhere requires a human override choosing a substitute" is sharpened to say assembly *omits* encounters whose monster is unresolved. Rationale in work item 5: emitting the dangling id would fail `validate_adventure` inside `GameSession.new` and make every draft with one unresolvable name unplayable — the milestone would be unreachable for any real module. Omission plus the `monster_unresolved` flag preserves the spec's intent (no invented substitutes; a human override is still the only way the encounter enters the adventure).
- **§ Pipeline, the geometry stage (stage 4):** "emit `Edge`s (open/door per the text)" becomes open-only for v1, with doors named as override/vision-pass territory. The door signal is not in the phase 1 extraction schema; adding a field now would strand every recorded fixture, and keyword inference over description prose misattributes doors too often to be worth flagging as truth.
- **§ Pipeline and § The workdir:** a parenthetical records that geometry is recomputed inside every assembly and has no cache — confirming the reading phase 0 pinned in `contracts/run.py` (whose "to be confirmed by phase 2" docstring note is removed).
- **§ Extraction report:** the example gains the additive module-level `flags` array (work item 1) so the spec stays the authority on the report shape.

Pinned reading recorded here (phase 1's pattern): phase 1's work item 1 said "phase 2's monsters stage owns the mapping onto osrlib's exactly-one-of rule". The mapping lands in assembly's encounter builder, not in the monsters stage: counts are per-encounter facts and `stages/monsters.json` is keyed per-name, so the monsters cache has nowhere to put them. The behavior phase 1 pinned (prefer dice when both are set; flag when neither is; demote unparseable dice to `count_note` plus a flag) is implemented verbatim in work item 5.

## Work items

### 1. Contracts and plumbing

- `contracts/stages.py` gains the monsters cache, `stages/monsters.json` = `MonsterResolutions`: `schema_version`, `resolutions: dict[str, MonsterResolution]`. Keys are **normalized names** (casefolded, internal whitespace collapsed, stripped — `normalize_monster_name` in work item 2), sorted ascending at write time for byte stability. Normalized keys are the point: the JN1 caches spell the same monster `"Zombies"` and `"zombies"` (likewise `"Giant Centipedes"`/`"giant centipedes"`), one resolution must serve both, and phase 3's override matching (`overrides.yaml` `monsters:` keys — the spec's own example is lowercase `"hobgoblin chieftain"`) normalizes the same way.
    - `MonsterResolution`: `template_id: str | None`, `method: Literal["exact", "alias", "fuzzy", "llm", "unresolved"]`, with a model validator pinning `template_id` non-null exactly when `method != "unresolved"`. Frozen, `extra="forbid"`, `schema_version` only — all per the phase 1 cache conventions (usage and model identity live in `run.json` only).
- `contracts/report.py`: `ExtractionReport` gains `flags: tuple[FlagString, ...] = ()` — module-scope flags with no per-area home (a defaulted adventure title, a defaulted town name; work item 5). Additive within the schema version; the spec's report example gains the field (spec impact above). No new `Flag` members: the vocabulary is a published contract UIs badge on, and every phase 2 condition fits an existing prefix with a detail string.
- `workdir.py`: property `monsters_json` → `stages/monsters.json`; method `preview_svg(dungeon_id: str, level_number: int)` → `previews/<dungeon>.<level>.svg` (filename unambiguity again guaranteed by the slug alphabet). `area_caches()`'s charter note is corrected and the clearing rule extended: **`survey()` (in its on-success clear) and `content()` (in its upfront clear) now also unlink `stages/monsters.json`** — a re-run of either upstream stage can change the encounter-name population, orphaning old resolutions exactly as it orphans old area caches.
- `settings.py` gains the two knobs the spec names for this stage, both consumed by work item 2:
    - `monster_fuzzy_threshold: float = 0.85` (gt 0, le 1). The value is pinned against measured catalog pairs, not taste: it accepts the observed true matches `normal man` → `Normal Human` (0.909) and `yellow mold` → `Yellow Mould` (0.957) and rejects the observed false neighbours `giant bee` → `Giant Bat` (0.778) and `gray jelly` → `Ochre Jelly` (0.667). The asymmetry justifying a high bar: a false accept silently substitutes the wrong monster into a playable draft; a false reject merely routes the name to the LLM tier, which sees the right candidates anyway.
    - `monster_llm_top_k: int = 8` (ge 1) — candidates offered per name in the LLM tier; far below the proven 512-value enum budget even summed across a whole module's unresolved names.
- Two stale docstrings the plan supersedes are corrected alongside: `contracts/run.py`'s "reading to be confirmed by phase 2" note (the geometry reading is confirmed, work item 5) and `contracts/stages.py`'s `AreaEncounter` docstring, which says "phase 2's monsters stage owns the mapping onto osrlib's exactly-one-of rule" — the mapping lands in assembly's encounter builder (the pinned reading above), and the docstring now points there.
- No new error types. Provider failures in the LLM tier propagate as `ProviderError`/`SchemaValidationError`; calling a stage over an incomplete upstream stage stays stdlib `ValueError` (phase 0's rule); and a *deterministic* assembly failure over validated caches is a code bug (assertion), not a runtime condition — `validate_adventure` findings are data, and they land in the report (work item 5).

### 2. Monster resolution — `monsters.py` (stage 3)

The resolution population, pinned: **the union of encounter monster names across every `stages/areas.*.json` cache** — the names assembly must map — not `survey.monster_names`. The survey list is a document-wide superset that includes wandering-table entries and townsfolk (JN1's includes `Banker`, `Merchant`, `Riding Horse`); resolving those would burn LLM budget on names no keyed encounter references and emit `monster_unresolved` flags nobody can act on. Report flags must be actionable.

`normalize_monster_name(text: str) -> str`: casefold, collapse every internal whitespace run to one space, strip. Not the slug function — override keys keep their spaces (`"hobgoblin chieftain"`).

The catalog's naming is irregular, verified against osrlib 1.1.0's 233 templates, and the matcher must not assume a rule: ids are mostly `descriptor_noun` (`giant_rat`, `fire_beetle`) but some names are comma-inverted (`Centipede, Giant` → id `centipede_giant`; likewise `Shrew, Giant`, `Toad, Giant`, `Weasel, Giant`), some are compound (`Owl Bear` for what modules print as `owlbear`), and the plain animals hide behind qualifiers (`Normal Wolf`, `Normal Human` — there is no `wolf` id). So each template contributes a set of **match forms**: its normalized name, the comma-inverted rearrangement (`"centipede, giant"` → `"giant centipede"`), its id with separators as spaces (`"owl_bear"` → `"owl bear"`), and the separator-squashed spellings of all of those (`"owlbear"`). Each extracted name contributes its normalized form, its naive singulars (strip `-s`, strip `-es` — the caches really do say `giant rats`, `stirges`, `giant centipedes`), and their squashed spellings.

The four tiers, exactly the spec's, each consulted only when the previous one misses:

1. **Exact:** any extracted-name form equals any template match form. Deterministic and total — covers minimod's whole population (`giant rats` → `giant_rat`, `giant centipedes` → `centipede_giant` via the comma-inverted form, `goblin`, `skeleton`, `stirges` → `stirge`).
2. **Alias:** a curated `MONSTER_ALIASES: dict[str, str]` (normalized name → template id) in `monsters.py`. Entry rule: only names observed in a real module run, each entry carrying a source comment (e.g. `wolf` → `normal_wolf`, observed in JN1; irregular plurals like `wolves` land here too when observed). The table is seeded during this phase's milestone work — **complete before the JN1 recording session, because the LLM request contains exactly the names tiers 1–3 left unresolved, so any later alias edit covering a JN1 name changes the request fingerprint and strands the fixture** (work item 7 documents the coupling; re-recording is the remedy when the table grows with the eval corpus).
3. **Fuzzy:** best `difflib.SequenceMatcher` ratio between the extracted name's normalized form and every template match form. Auto-accept only when the best score is at least `monster_fuzzy_threshold` *and* the best template is unique at that score; a tie goes to the LLM tier rather than a coin flip.
4. **LLM:** one request for all still-unresolved names. Per name, the top `monster_llm_top_k` templates are offered as candidates — **ordered by fuzzy score descending, ties broken by template id ascending, the top-k cut taken after that ordering** (candidate identity and order are part of the frozen fixture's request fingerprint, so they must be deterministic) — as id plus printed name, so the model sees `killer_bee (Killer Bee)` when judging `giant bee`. Schema: one required object property per unresolved name **in sorted-name order** (the proven dynamic-schema pattern from content's key enums; the prompt lists names and candidates in the same order), each `{"template_id": ...}` with `enum` of exactly that name's candidate ids plus `null`, `additionalProperties: false` throughout. The system prompt pins the judgment: pick the template a referee would treat as the same creature under another name (edition synonyms, spelling variants, singular/plural); answer null when no candidate is that creature — never pick a merely similar monster. `null` → `method: "unresolved"`. Request tag: `monsters` — the request carries no page images, only text, which is what makes its fixture replay-grade (work item 7).

`build_monsters_request(names_with_candidates) -> ModelRequest` is public and pure, like `build_survey_request`/`build_batch_request` — the runner and fixture tests must build fingerprint-identical requests without duplicating prompt code.

The stage function `monsters(workdir: Workdir, provider: ModelProvider) -> MonsterResolutions`: require content `completed`, the survey cache, and every level's area cache present (else `ValueError`); under `track_stage(Stage.MONSTERS)`, collect and normalize the name population, run tiers 1–3, call the provider **only if names remain** (a fully deterministic resolution makes no model call — pinned and tested with a provider that raises on use), write `stages/monsters.json` (single atomic artifact; no pre-clearing needed), return the cache model. An empty name population writes an empty cache and completes.

### 3. Geometry synthesis — `geometry.py` (stage 4)

Deterministic code only; recomputed inside every assembly (no cache), completing the `run.json` `geometry` entry — the phase 0 reading, hereby confirmed. Input: the survey index plus every level's content cache; output: a frozen per-level result (area cell clusters, the `edges` dict, entrance, transitions, width/height) that work item 5 folds into `LevelSpec`s.

- **Room size.** Parse stated dimensions from each area's description — the first match of a feet-by-feet pattern (`30' x 40'`, `30 × 40'`, `30 feet by 40 feet`; at least one unit marker required so bare number pairs don't match). Cells: `ceil(feet / 10)` per axis (the spec's 10' cell), minimum 1; first number is width (east–west), second is height. No match → the 2×2 default. The committed evidence, honestly split: every minimod area states dimensions (`A 20' x 30' chamber…`), while JN1's extracted descriptions carry exactly one parseable pair across 137 areas (halfling-home h3's "15 x 17 foot dining room" — the module's other feet references are ceiling heights and ramp lengths), so expect JN1's previews to be dominated by 2×2 default rooms; the parser earns its keep on dimension-stating modules and the correction loop owns the rest.
- **The connection graph.** Per level, resolve each connection's `to_key`: exact canonical key on the level, else `canonical_slug(to_key)` match on the level. Unresolved locally with direction `up`/`down` → try the sibling levels of the same dungeon (exact, then slug) — a hit is a level transition (below). Still unresolved → the connection is dropped from geometry and assembly flags the owning area `connection_ambiguous` (work item 5). Duplicate mentions collapse to one undirected edge whose direction is the first stated in survey order (a reverse-only mention is inverted); self-connections are dropped. A resolved `up`/`down` *within* one level — real data: minimod's pit-room, `5 → 6 down` — is an ordinary intra-level connection placed as direction-unknown, since a 2D grid has no third axis.
- **Anchors and the entrance.** The dungeon's entrance lives on its lowest-numbered level; the entrance area is that level's first area in survey order — the convention every observed module follows (JN1 lair A's area 1 is "Gaping entranceway…", minimod's area 1 is "Broken steps descend from the surface"); `LevelSpec.entrance` is the entrance area's first cell. Non-entrance levels BFS from their own anchor: the transition-target area lowest in survey order, else the level's first area.
- **Placement.** BFS from the anchor over the connection graph — a FIFO queue, each area's neighbours enqueued in the survey order of the *target* area, so visit order (and therefore every collision and every golden byte) is deterministic. The anchor's rectangle sits at the origin. Each BFS edge places the child room adjacent to its parent in the stated compass direction (grid convention: x east, y south, so north is −y), or in the first free direction of the pinned order north, east, south, west when the direction is unknown/vertical. Candidate placements are enumerated in a pinned deterministic order — increasing corridor length from 1, then lateral offset 0, +1, −1, +2, … perpendicular to the direction — and the first candidate whose room cells *and* corridor cells are all unoccupied wins (the spec's "shift on collision"). The corridor is a 1-cell-wide, direction-first Manhattan path from the parent's facing-side midpoint cell to the child's (an even-length side has two middle cells; the one with the smaller coordinate is the midpoint); corridor cells belong to no area, which is exactly osrlib's corridor semantics ("cells not in any area are corridor").
- **Edges — the impassability hazard, defused.** Emit `Edge(kind=open)` between every orthogonally adjacent pair of same-area cells, along every corridor path, and at both room–corridor junctions, using osrlib's canonical `edge_key` form (north/west keys only — a cell's east edge is stored as its eastern neighbour's west edge; the overrides contract already documents this canonicalization hazard). All synthesized edges are `open` in v1 (spec impact above); the preview renderer still draws doors because phase 3's override re-renders will contain them.
- **Disconnected components.** An extracted graph can be disconnected (missed connections). Components are ordered by the survey position of each component's first area (the anchor's component first); each later component joins by a synthetic direction-unknown link from its first area to the previous component's first area, so placement stays one BFS and the level stays traversable; every area of a synthetic-linked component is flagged `connection_ambiguous` with detail `not connected to the entrance in the extracted graph` (work item 5). A corridor is synthesized geometry — already flagged as such everywhere — not invented content.
- **Normalization.** After placement, translate all cells so the minimum coordinate is (0, 0); `width`/`height` are the bounding box; every cells tuple is sorted row-major (y, then x) — determinism the byte-stability tests rely on.
- **Transitions.** Each cross-level link becomes a reciprocal `TransitionSpec` pair: `stairs_up` at the source area's first cell when the stated direction is `up` (`stairs_down` for `down`), targeting the other area's first cell on its level, with the mirror-image spec on the far side; `to_facing` is pinned `north` (arbitrary but deterministic; it only sets post-arrival facing). Both directions extracted — JN1's manor states `85 → 98 up` *and* `98 → 85 down` — deduplicate to one pair.
- **Postconditions, asserted in code and pinned in tests** (these are what make the spec's "the output always satisfies osrlib's structural rules" true by construction, since `validate_adventure`'s geometry checks are a subset): every area has ≥ 1 cell; no two areas share a cell and no corridor crosses a room; all cells, the entrance, and every transition position are in bounds; the entrance level has `entrance` set; every area on a level is reachable from its anchor through `open` edges; every transition's target level and cells exist.

### 4. SVG previews — `previews.py`

One public function, `render_level_svg(dungeon_id, level_spec) -> str`, pure string assembly with no dependencies and no timestamps: a fixed-size cell grid (24 px), area cells filled with the area key labelled at its first cell, corridor cells in a lighter fill, walls drawn as solid strokes wherever the edge map says wall (including the implicit boundary), doors as a distinct tick (v1 synthesis emits none, but phase 3's override re-renders will — the renderer supports the full `Edge` model from birth), the entrance marked, and transitions glyphed (▲ stairs up, ▼ down). Assembly writes one file per level to `previews/<dungeon>.<level>.svg`; `osrforge preview` (work item 6) regenerates previews alone. Byte-stable across runs — previews are one of the three pure artifacts, and the phase 0 pin ("no timestamps in pure artifacts, tested once report production exists") is enforced by work item 9's guard test.

### 5. Assembly and report — `assemble.py` (stage 5)

`assemble(workdir: Path) -> AssembleResult` (frozen: `adventure: Adventure`, `report: ExtractionReport`) — the spec's "Adventure + report, pure". Requires monsters `completed` and all caches present (else `ValueError`). Two sequential trackings: `track_stage(Stage.GEOMETRY)` around synthesis, then `track_stage(Stage.ASSEMBLE)` around the build, validation, and artifact writes — so a failure in the second leaves an honest `geometry: completed`. Neither stage touches a provider; usage stays zero and `run.json`'s provider identity is untouched. Overrides are **not** read in phase 2 — application is phase 3's charter (both prior plans pin this); until then assembly is a pure function of the stage caches alone, and `report.json`'s per-area `overridden` stays empty.

The adventure build, top down:

- `Adventure.name` = the survey title; empty → `"Untitled module"` plus module flag `low_confidence:module title unstated`. `description` = `""` (nothing extracted claims to be a module description; inventing one from hooks would be fabrication). `hooks` = the survey's, verbatim.
- `TownSpec`: `name` = the survey's; empty → `"Town"` plus module flag `low_confidence:town name unstated` (the default-plus-flag treatment `contracts/stages.py` pinned in phase 1). `description` from the survey; `services` empty (not extracted); `travel_turns` empty — osrlib treats an absent entry as zero travel time, the value is nowhere in the extraction, and the overrides contract already owns `travel_turns` for humans who care.
- **Areas**, one `AreaSpec` per survey area in survey order: `id` = the canonical key, `name` from the survey, `cells` from geometry. An area missing from its level's content cache (the model skipped it twice, or the level had no pages) becomes the placeholder phase 1 pinned: empty description, no encounter/trap/treasure/features, confidence 0.0, flagged `low_confidence:not extracted`.
- **Encounters — the unresolved-omission decision.** All of an area's cache encounters merge into one `KeyedEncounter` (osrlib's model: one per area, many `KeyedMonster` groups). Per cache encounter: look up the normalized name in `stages/monsters.json` (a missing key is a stale-cache `ValueError` — convert's ordering makes it unreachable); **resolved** → a `KeyedMonster` with the count mapping below; **unresolved** → no `KeyedMonster`, and the area is flagged `monster_unresolved:<normalized name>`. An area whose every encounter is unresolved gets `encounter=None`. Why omission instead of the dangling id: `GameSession.new` validates first, so one dangling id makes the whole draft unplayable — no TUI, no milestone, for any module with a single bespoke monster (JN1 has several; DCC #81 is mostly bespoke). Omission keeps the draft playable while the flag plus the report's unresolved list drive the human to choose a substitute through overrides — which remains the only path for that encounter to enter the adventure, preserving the spec's no-invented-substitutes rule. (Spec impact applied, above.)
- **Counts** (the phase 1 pins, implemented here): `count_dice` preferred when both it and `count_fixed` are set; a `count_dice` osrlib's `parse` rejects is discarded in memory (the cache is never rewritten; purity) and flagged `low_confidence:unparseable count for <normalized name>` — the encounter falls back to `count_fixed` when set, else follows the neither-set rule below (all of it unreachable through the schema's `DICE_PATTERN`, which is a strict subset of osrlib's grammar — defense in depth, tested directly); neither dice nor fixed → `count_fixed=1` with flag `low_confidence:count unstated for <normalized name>` — one is the minimum the module's naming of the monster implies, and the flag routes a human to the cache's `count_note`.
- **Traps:** the cache's free-text `trap` becomes `TrapSpec(kind="room", trigger="enter", affects="triggerer", effect=TrapEffect(manual=<text>))`. `manual` is osrlib's referee-adjudicated escape hatch, and `kind="room"` is mandatory on an `AreaSpec` trap (osrlib validates exactly that). Mechanical effects are override territory.
- **Features:** each cache feature string becomes `FeatureSpec(id=f"{key}-f{n}", kind="custom", description=<text>)` (1-based n). Ids are unique per level because area keys are; the reserved id `pile` is unreachable by construction (`-f` suffix). `cell=None` binds the feature to its whole area — osrlib's documented semantics for area-listed features.
- **Treasure**, parsed per cache string through a pinned, conservative grammar (tried in order; first match wins; anything else is flagged `treasure_unparsed:<the string>` and omitted from the draft — the standing escape both prior plans reserve for exactly this):
    1. Contains dice notation or the words `each`/`per` → unparsed (per-monster and conditional treasure — JN1's `Each orc has 1d6 sp.` — cannot be a fixed cache).
    2. `<thing> worth <N> gp` → a `ValuableSpec`: `name` is the text before ` worth` with a leading article (`a`/`an`/`the`) stripped; `kind="gem"` exactly when that text contains the word `gem`/`gems` (word-boundary, case-insensitive), else `"jewellery"` (minimod's `silver locket worth 25 gp` → jewellery). The narrow gem lexicon is deliberate: a diamond classified as jewellery is harmless — both kinds carry identical `value_gp` and XP semantics — while a broad stone-name list invites misparses.
    3. Money references `<N> <cp|sp|ep|gp|pp>` with no digits outside them → `Coins` summed per denomination (minimod's `120 sp`, `pot of 200 gp`).
    4. `treasure type <A–V>` → the area's `AreaTreasureSpec(letters=…)` — osrlib's generated-treasure channel, distinct from literal amounts.
    Parsed coins and valuables aggregate into one `FeatureSpec(id=f"{key}-treasure", kind="treasure_cache", coins=…, valuables=…)` per area — the osrlib shape that makes treasure takeable (`TakeTreasure`) and XP-bearing (coins and valuables are the *only* treasure XP source; magic items can't be authored into a cache at all, so `potion of healing` is honestly unparsed). `AreaTreasureSpec` is letters-xor-unguarded and phase 2 never emits `unguarded`.
- **Connections** produce flags, not content: `connection_ambiguous:unresolved target <to_key>` for a target geometry couldn't resolve, and `connection_ambiguous:unknown direction to <key>` for a resolved connection whose direction was `unknown` — the deterministic flag derivation phase 1 pinned for exactly this field.

The report, regenerated on every assembly: `module` (survey title verbatim — the report records facts, defaults live only in the adventure — and `run.json`'s page count); one `AreaReport` per survey area in survey order — address string, the content cache's `source_pages` (the survey's for placeholders), confidence (0.0 for placeholders), and flags in pinned emission order (`geometry_synthesized` first — every area carries it in v1 — then `low_confidence`, `monster_unresolved`, `connection_ambiguous`, `treasure_unparsed`, each group in derivation order); `monsters` (resolved = count of resolved names in the cache, unresolved = the sorted unresolved names); `usage` = the sum of the survey, content, and monsters stages' usage from `run.json`; the module-level `flags` from the defaults above. Validation: run `validate_adventure(adventure, load_monsters(), load_equipment())`, catch `ContentValidationError`, and record `passed`/`errors` (the exception's per-finding lines, header stripped). By construction it should pass — geometry's postconditions and encounter omission cover every check — but the report records what the gate actually said, never an assumption.

Artifacts, all through `write_json_artifact` and the pinned byte format: `adventure.json` = `stamp_document("adventure", adventure.model_dump(mode="json"))` — osrlib's envelope, currently schema_version 2, embedding osrlib's `engine_version`; that embedding is a feature, not a wart: an osrlib upgrade changes the goldens and fails CI loudly, which *is* the spec's golden-fixture compatibility gate (work item 7). Then `report.json`, then every level's preview. Byte-stability across two identical `assemble()` calls is asserted by work item 9, and no artifact may contain a timestamp (the phase 0 pin, now enforceable).

### 6. `convert()`, the CLI, and the façade — `convert.py`, `cli.py`

- `convert(pdf_path: Path, workdir: Path, provider: ModelProvider, settings: ConversionSettings | None = None, on_progress: OnProgress | None = None) -> ConversionResult` (frozen: `run: RunMeta`, `adventure`, `report`): `preprocess → survey → content → monsters → assemble`, each stage already writing its own `run.json` status; a stage failure propagates after its `failed` status is written, keeping everything upstream (the spec's resume story, with `rerun` arriving in phase 3). `on_progress` receives a frozen `StageEvent(stage: Stage, status: StageState, usage: TokenUsage | None)` — emitted `running` before each stage and `completed`/`failed` (with that stage's usage from `run.json`) after — the stage-transitions-and-usage stream the spec promises hosts. Events cover the five chain steps `convert` drives; the `geometry` stage tracked inside `assemble()` gets no separate event (from a host's perspective it is an implementation detail of assembly — its status is still readable in `run.json`).
- `cli.py` with argparse (house scale; no new dependency) and `[project.scripts] osrforge = "osrforge.cli:main"`. Phase 2 subcommands, exactly the spec's syntax: `osrforge convert <module.pdf> [--workdir DIR] [--provider foundry]` (workdir defaults to `./<pdf-stem>.forge`; `--provider` has the single choice `foundry`, built from `FoundrySettings.from_env()`), `osrforge assemble [--workdir DIR]`, and `osrforge preview [--workdir DIR]` (both defaulting `--workdir` to the current directory, so running inside a workdir just works; `preview` re-runs geometry synthesis and rewrites `previews/` only, touching neither the other artifacts nor `run.json`). `check`, `rerun`, and `estimate` are phase 3's. `OsrForgeError` (and pydantic validation errors from malformed workdirs) render as a one-line message and exit code 1; tracebacks are for bugs.
- The façade grows to the spec's promise as of this phase: `__init__` re-exports `convert` and `assemble` beside `ConversionSettings` (`check` and `estimate` join in phase 3). Result types stay importable from their home modules — one home per symbol.
