# Phase 3 plan — the correction loop

Implementation plan for phase 3 of [the osr-forge spec](spec.md). Phase 3 delivers the human correction loop the spec promises: overrides *application* (phase 0 shipped the models; nothing reads them yet), `rerun`/resume, the playability lint with its smoke delve, `estimate`, and CLI polish. The roadmap milestone: fix a bad draft to publishable entirely through `overrides.yaml`. Every decision below rests on the osrlib 1.1.0 surfaces verified against the checkout at `~/repos/osrlib-python`, the committed phase 1/2 caches and goldens, and the measured costs in `docs/foundry-capabilities.md` plus the four recorded full-module runs; the hazards that would silently kill a correction are woven into the work items that defuse them.

Three osrlib facts shape everything and are worth stating before the work items:

- **A wrongly keyed edge is silently a wall.** `LevelSpec.edges` is consulted through `edges.get(edge_key(...), Edge(kind=wall))`, and `edge_key` stores only `north`/`west` keys (a cell's east edge is its eastern neighbour's west edge; its south edge the southern neighbour's north). A door override written as `"5,2:east"` and stored verbatim would never be consulted — the door simply wouldn't exist, with no error anywhere. Application must canonicalize through osrlib's own `edge_key`, and the lint must flag any in-draft edge key that can never take effect.
- **osrlib validates content references, never geometry connectivity.** `validate_adventure` checks cells/entrances/transitions in bounds and ids resolving; it never touches `edges` — a fully sealed entrance, an unreachable area, or a door authored on an out-of-bounds key all pass, and `LevelSpec` itself has no validators at all. Phase 2's synthesis guaranteed connectivity by construction; the moment humans override geometry, that guarantee is gone and only the playability lint stands between a green `validate_adventure` and an unplayable draft.
- **Doors gate movement through a small state machine.** A closed door rejects `MoveParty` (`exploration.move.blocked`); `OpenDoor(direction)` opens a plain door deterministically and rejects `stuck`/`locked`; `stuck` needs a probabilistic `ForceDoor`, `locked` needs a thief's `PickLock`, and an undiscovered `secret` door is a wall to every command until a probabilistic `Search` finds it. Party-opened doors swing shut once the party moves away. The smoke delve therefore walks the *deterministically traversable* subgraph — open edges plus plain doors — and the static checks own the rest.

## Scope

In scope:

- Contracts and plumbing: `LintFinding` and the additive `ExtractionReport.findings` field (the phase 0 plan's named deferral), `OverrideError`, duplicate-key-rejecting YAML loading (phase 0's named gap), and the `blank_page_renders` settings knob with its preprocess behavior and `page_unreadable` flags (the phase 1 content-filter hazard, promoted from "recorded, not promised")
- `overrides.py` — application of every v1 override kind inside assembly, the `overridden`/flag semantics in the report, and override-aware previews
- `check.py` — the playability lint: consumer-grade artifact loading, `validate_adventure`, the static graph checks, the smoke delve, findings merged into `report.json`
- `rerun()` — resume/re-run from any stage through assemble, with the settings-update channel (`--set`) and the knob→stage drift guard
- `estimate.py` — preprocess-only cost estimation with pinned heuristics calibrated against the four recorded full-module runs
- CLI: `check`, `rerun`, `estimate` subcommands, `--set` on `convert`/`rerun`, `--version`, the commented `overrides.yaml` template, and the façade growing to the spec's full library API (`check`, `estimate`)
- `tools/extract/` extension for override-aware goldens; the JN1 correction session and its corrected goldens
- The milestone: the committed JN1 `overrides.yaml` that takes the real draft to publishable, byte-gated in CI, plus the session amendment
- Tests for all of the above, green in CI with no network

Out of scope (later phases): survey chunking past the guard, the eval corpus and scoring harness, docs site, and PyPI release (phase 4); door/geometry *extraction* from map images (the spec's vision-first open question — doors arrive through overrides, which this phase makes real); custom `MonsterTemplate` emission, non-PDF inputs, OCR fallback (the spec's standing open questions); full automated playthroughs (the spec names them future work — the delve stays a bounded walk); level add/remove through overrides (the spec's v1 override kinds are monster/area/geometry/town/module — a whole missing level is a re-extraction problem, not a correction); an overrides version field (phase 0's decision stands: strict unknown-key rejection makes a future revision detectable).

**Constraint carried from phase 2: no prompt, schema, or alias-table edits.** Phase 3 touches no extraction stage and no `MONSTER_ALIASES` entry — any such edit changes request fingerprints and strands the committed minimod and JN1 fixtures. Everything the correction loop needs is downstream of the caches. (A monster override supersedes a wrong resolution at assembly; it never re-records the monsters fixture.)

Spec impacts, applied with this PR so the spec stays the authority:

- **§ CLI:** the `rerun` line becomes "re-run one stage — and everything downstream of it — from cached upstream outputs" (the pinned resume reading, work item 4); `estimate` gains `[--workdir DIR]` (the licensing invariant forces preprocess output into the user's workdir, so the command must accept one); `convert` and `rerun` gain the repeatable `[--set KEY=VALUE]` settings channel (work items 4 and 6 — without it neither `blank_page_renders` nor any other knob is reachable from the CLI).
- **§ Overrides:** three application rules join the contract: monster override keys match extracted names under the same normalization the monsters stage uses; every override entry must take effect or assembly fails with a named error (no silent no-ops in a correction file); an area *add* must carry `name`, `description`, and a geometry override supplying its cells. Duplicate YAML keys are rejected at load (closing phase 0's recorded gap).
- **§ Extraction report:** the example gains the additive `findings` array (work item 1) — the structured lint findings the spec already promises in § Validation and playability checks.
- **§ Validation and playability checks:** the finding ids and severities are enumerated (work item 3), and the smoke delve's shape is pinned: per dungeon, entrance level, deterministic-subgraph path, encounters evaded.
- **§ Configuration:** the knob list gains `blank_page_renders`; a sentence records that changing settings on an existing workdir goes through `rerun --set`, guarded by the knob→stage map.

## Work items

### 1. Contracts and plumbing

- `contracts/report.py` gains the lint vocabulary, additive within the schema version:
    - `LintCheck` — a `StrEnum` of the finding ids: `edge_invalid`, `area_unreachable`, `orphan_cell`, `secret_only_access`, `transition_unpaired`, `delve_blocked`, `delve_incomplete`. Like `Flag`, a published vocabulary UIs badge on; growing it is additive, renaming is a schema-version event.
    - `LintFinding` — frozen, `extra="forbid"`: `id: LintCheck`, `severity: Literal["error", "warning"]`, `location: str` (an area or level address where one applies; the dungeon id for delve findings), `message: str`. Severity stays a field rather than a function of the id — the id→severity table is the *producer's* pin (work item 3), so the contract needn't change if a check's severity is ever re-judged.
    - `ExtractionReport.findings: tuple[LintFinding, ...] = ()` — empty from `assemble()` (stale lint about a changed draft is worse than none; re-assembly wipes findings by design), populated by `check()`. The spec's report example gains the field.
- `errors.py` gains `OverrideError(OsrForgeError)`: an override entry that cannot take effect — unknown monster name, unknown area or level address that isn't a well-formed add, contradictory entries, an edge collision after canonicalization. Application failures are runtime failures of the correction work itself, not programmer misuse; the CLI already renders `OsrForgeError` as one line. The division of labor, pinned: **addressing errors are loud (`OverrideError`); content validity flows to the report.** An override whose `template_id` doesn't exist in the catalog *does* take effect — the dangling id lands in the draft and `validate_adventure` reports it in `report.json`, which is exactly the loop the human is already in. Pre-validating content at application time would build a second validator beside osrlib's — the thing AGENTS.md forbids.
- `contracts/overrides.py`: `load_overrides` rejects duplicate YAML mapping keys (a `SafeLoader` subclass whose mapping constructor raises on a repeated key) — phase 0 recorded silent-last-wins as the known gap for exactly this phase; in a human-edited correction file a duplicate key means two contradictory corrections, and one of them silently losing is the worst outcome. A duplicate key raises `OverrideError` naming the key and line (it is a correction that cannot take effect — work item 2's charter — and it must render as the CLI's one-line message, which `yaml.YAMLError` would not); syntactically invalid YAML keeps raising `yaml.YAMLError`, which the CLI learns to render (work item 6).
- `settings.py` gains `blank_page_renders: tuple[int, ...] = ()` (each ≥ 1): page numbers whose *renders* are emitted as blank white PNGs at the page's normal pixel size, text layer still extracted — the exact manual workaround the B3 verification run proved out when Azure's image content-safety filter rejected a page wholesale (phase 1 amendment). `preprocess.py` implements the blanking; pages out of range are rejected by preprocess (`PdfError` — the knob names pages, and a reference to page 99 of a 48-page module is a wrong-file-shaped mistake). Assembly (work item 2) emits one module-level flag per blanked page — `page_unreadable:page <N> render blanked` — giving the report vocabulary's last unproduced flag its producer; the derivation is pure because the settings echo in `run.json` is already an assembly input.
- No new workdir paths: overrides application reads the existing `overrides_yaml`, `check` reads and rewrites the existing artifacts, and `estimate` reuses the whole preprocess layout.

### 2. Overrides application — `overrides.py`

The application module (contract models stay in `contracts/overrides.py`; application is assembly-side code, the same split as report contracts vs. report production). `assemble()` loads `overrides.yaml` once and threads it through; assembly stays pure — the artifacts become a deterministic function of the stage caches *plus the overrides file*, which is the spec's core guarantee verbatim.

**The anti-silent-no-op rule, pinned:** every entry in a correction file must take effect or assembly fails with an `OverrideError` naming the entry. A typo'd monster name or area address that silently did nothing would let a human believe a correction landed when it didn't — in a file whose whole purpose is reproducible correction, that is the one unforgivable failure mode.

Application points, in build order:

- **Monsters.** Override keys are matched to extracted names under `normalize_monster_name` (the spec's own example key, `"hobgoblin chieftain"`, is already normalized; a human who types `"Hobgoblin  Chieftain"` still hits it). Two keys normalizing to the same name → `OverrideError` (contradiction); a key matching no name in the monsters cache → `OverrideError` listing the cache's unresolved names (the likeliest targets). A matched override replaces the cached resolution before encounter building: the stand-in (or omission) never happens, no `monster_unresolved` flag is emitted, and the report's `monsters` summary counts the name resolved. Per-area `overridden` is *not* marked — the correction is name-scoped, not area-scoped, and its visible home is the monsters summary plus the flag's disappearance. The monsters stage itself never reads overrides: a monster correction needs only `assemble`, not a model re-run.
- **Areas.** Per entry, `model_fields_set` distinguishes the three field semantics phase 0 pinned: absent = untouched, explicit `null` = cleared, value = replaced. An overridden field replaces the *cache-derived* value in `_build_area` before any parsing: an overridden `encounter` skips encounter building entirely (the osrlib `KeyedEncounter` payload is used verbatim), and an overridden `trap` skips the trap mapping. **The treasure grammar has two outputs and each override controls exactly one** — today's grammar feeds both `AreaSpec.treasure` (letters, or the best-effort `unguarded` fallback) and the parsed coins/valuables cache emitted as the `<key>-treasure` feature inside `features`, so the coupling is pinned rather than guessed: the `treasure` override replaces (or `null`-clears) only the `AreaSpec.treasure` slot, and the best-effort unguarded fallback never fires for an area whose `treasure` is overridden — the human's word is the last word; the `features` override replaces the area's *final* features tuple wholesale — the mapped `-f{n}` features *and* the parsed `-treasure` cache (partial merging of one osrlib field is ambiguous, and the osrlib payloads are embedded directly, so a human who wants literal treasure authors their own `treasure_cache` feature) — which also makes a feature-id collision with the parsed cache impossible by construction. The grammar runs only for the slots not overridden; when both are overridden it is skipped entirely. `remove: true` skips the `AreaSpec`; the area's placed cells remain and become corridor (osrlib's "cells in no area" semantics) — removal deletes content, not floor plan, and a human who wants the floor plan gone owns the geometry override for it. An entry addressing an area the survey doesn't have is an **add**: it must carry `name` and `description` (set and non-null) and a geometry override supplying its cells (an added area has no synthesized placement to inherit), else `OverrideError`; the add is appended after the level's survey areas in entry order. Contradictions are errors: `remove` combined with any replacement field, or `remove` combined with a geometry cells entry for the same area.
- **Geometry.** Applied to each `LevelGeometry` after synthesis, before `LevelSpec` construction. Level addresses must name survey levels (`OverrideError` otherwise — you cannot add a level, see scope). Per level: `areas` cells replace that area's cluster wholesale (the key must be a survey area or a valid add on that level); `entrance` replaces; `transitions`, when present, replace the level's tuple wholesale — partial merging of a list that comes in reciprocal pairs is ambiguous, and the overrides contract already models absent-means-untouched at the field level. `edges` are canonicalized through osrlib's own `edge_key` (parse `x,y:direction`, re-key east as the neighbour's west, south as the neighbour's north — the phase 0 hazard, defused where phase 0 said it would be) and merged over the synthesized map, override winning per canonical key; two override keys canonicalizing to the same edge → `OverrideError` (the spec's own `"5,2:east"` example becomes `"6,2:west"` — the canonicalization table test, work item 8). An override edge may be `kind: wall` — sealing a synthesized opening is a legitimate correction. Negative coordinates anywhere in a geometry override → `OverrideError` (the edge-key grammar already forbids them; cells get the same rule, since osrlib's `in_bounds` starts at 0 and a negative cell can never be valid). After application, `width`/`height` are recomputed as the bounding box of all final area cells and surviving corridor cells. **Synthesis postconditions are not re-asserted after overrides** — they guard code, and human input is a runtime condition, not a bug; what a bad geometry override breaks, `validate_adventure` and the lint report.
- **Town and module.** Field replacement on `TownSpec` and the adventure's name/description/hooks, applied where the defaults are today: an overridden name wins over both the extracted value and the default, and the corresponding `low_confidence:… unstated` module flag is not emitted (it describes the draft, and the draft is no longer defaulted).

**Flags describe the built draft — the unifying rule.** Because overrides replace *inputs to the build*, every flag falls out of the path actually taken: overridden encounter → no monster or count flags; `treasure_unparsed` is emitted only when the grammar actually ran (both `treasure` and `features` overridden → no parse, no flags; one overridden → the leftovers still describe strings the draft couldn't place); cells overridden → that area drops `geometry_synthesized` (the badge review UIs clear as corrections land); overridden title → no default flag. Connection flags (`connection_ambiguous`) persist regardless — they describe the extracted connection graph, which no override kind touches. Extraction facts also persist: `confidence` and `source_pages` stay the cache's (an added area reports `source_pages=()`, confidence 1.0 — it is human-authored, the one place a human statement *is* the extraction).

**`overridden` vocabulary, pinned:** per-area, the replaced field names (`name`, `description`, `encounter`, `trap`, `treasure`, `features`), `cells` for a geometry cells entry, `removed`, `added`. A removed area keeps its report entry (id, cache facts, `overridden: ["removed"]`, no flags — nothing was built); review UIs need the tombstone.

**Previews follow the draft.** The effective per-level geometry and area roster (post add/remove/cells/edges/entrance/transitions) is computed by one shared helper that both `build_draft` and `render_previews` consume, so `osrforge preview` shows corrected cells and — for the first time — real doors: the phase 2 renderer has drawn door ticks and secret-door variants "from birth" for exactly this moment.

### 3. Playability lint and the smoke delve — `check.py`

`check(workdir_path: Path) -> tuple[LintFinding, ...]` — the spec's `findings = check(workdir)`. Requires `adventure.json` and `report.json` (else `ValueError`). It loads the adventure exactly as a consumer does — `check_document` + `Adventure.model_validate` — so every `check` run also exercises the artifact contract, then runs the spec's two tiers and merges: the loaded report's `findings` are replaced (`model_copy`) and `report.json` rewritten through the pinned writer. Everything is deterministic (the delve seed is a module constant), so the purity guarantee extends: `assemble && check` twice is byte-identical, and the no-timestamp guard covers the rewritten report.

- **Tier 1 — content validation:** `validate_adventure(adventure, load_monsters(), load_equipment())`, run directly on the loaded adventure (the spec's own definition of the tier; by purity its outcome matches the report's `validation`, which is not rewritten). Its result feeds the CLI verdict, not `findings` — osrlib's errors are strings with an existing report home, and duplicating them as findings would make two sources of truth.
- **Tier 2 — static graph checks.** Three graph flavors over each level's edge map, pinned once: *inclusive* (open edges plus every door, any state — what a party can traverse in principle), *non-secret* (secret-door edges become walls), *deterministic* (open edges plus plain doors only — no stuck, locked, or secret; what a scripted walk can traverse without probabilistic commands). Transitions are directed edges between cells (a `TransitionSpec` points one way; chutes and trapdoors never pair back). The checks, each with its pinned id and severity:
    - `edge_invalid` (error): an edge-map key that is malformed, non-canonical (`south`/`east` form), or has either incident cell out of bounds. osrlib silently ignores such keys (the wall-default hazard above), so a door the author believes exists doesn't — the single most dangerous silent failure in the whole correction loop.
    - `area_unreachable` (error): BFS over the inclusive graph plus directed transitions, seeded from *every* dungeon's entrance cell simultaneously (the party can enter any dungeon from town); any keyed area none of whose cells are reached. Covers the sealed entrance as the degenerate case.
    - `orphan_cell` (warning): a cell in no area that carries at least one non-wall edge (the preview renderer's own definition of a corridor cell — in osrlib every blank bounding-box cell is technically "corridor", and flagging those would drown the report) yet is never reached by the inclusive BFS — it renders on the map but can never be visited. Warning, not error: synthesis never produces one; only an override can, and it may be deliberate map dressing.
    - `secret_only_access` (warning): an area reachable in the inclusive graph but not in the non-secret graph — the spec's "at least one non-secret path" check, warning by the spec's own ruling (some modules intend it).
    - `transition_unpaired` (warning): a `stairs_up`/`stairs_down` whose target cell has no reciprocal transition back to the source — players who descend can't climb back. Trapdoors and chutes are exempt (one-way by osrlib's design).
- **Tier 2, dynamic — the smoke delve**, per dungeon, bounded and deterministic: a throwaway seeded party (four level-1 characters via osrlib's `create_character`, pinned seed constant), `GameSession.new` (which re-validates — a draft failing tier 1 skips the delve entirely, its findings would be noise), `EnterDungeon`. Compute the BFS path over the *deterministic* subgraph from the entrance cell to the farthest reachable cell on the entrance level, then walk it reactively: `MoveParty` per step; on an `exploration.move.blocked` rejection where the level spec has a door edge, `OpenDoor(direction)` and retry once — the reactive pattern sidesteps `starts_open` bookkeeping and swing-shut entirely; any other rejection of a step the graph deemed passable is `delve_blocked` (error — the static model and the engine disagree, which is a real defect by definition). After any command, while the session is in encounter mode, disengage with `Evade()` — and when the evasion draws pursuit, drive the pursuit rounds with `Wait` until it resolves (in osrlib a pursued evasion leaves the session in encounter mode and further `Evade` commands reject `already_evading`; `Wait` is what advances pursuit — the phase 2 smoke never met this because its evasions succeeded outright) — under a per-encounter budget of 20 commands (the phase 2 milestone's lesson: entrance cells hold keyed encounters — minimod's area 1 — and default wandering checks fire under any seed); if battle opens (an `attacks` stance pre-empts commands, and pursuit can end in battle) or the budget exhausts, the dungeon's delve ends with `delve_incomplete` (warning — geometry up to that point played; module difficulty is not a geometry defect). If a transition is reachable on the deterministic subgraph, walk to it and `UseStairs` once — the one dynamic transition exercise. Hard cap: 200 commands per dungeon. A clean delve emits nothing.
- **CLI `osrforge check [--workdir DIR]`:** prints the validation verdict and one line per finding (`severity id location message`); exit 1 exactly when validation failed or any error-severity finding exists, else 0 — warnings don't break the `assemble && check` loop. A clean exit is the *necessary* half of the milestone's "publishable" bar; the full definition has one home, work item 7.

### 4. `rerun` and resume — `convert.py`

`rerun(workdir: Path, stage: Stage, provider: ModelProvider | None = None, settings_updates: Mapping[str, object] | None = None, on_progress: OnProgress | None = None) -> ConversionResult`, beside `convert` (whose chain it shares — one chain runner, `convert` calling it from `preprocess`).

- **Pinned reading: `rerun <stage>` re-runs the named stage *and everything downstream* through assemble.** The spec's CLI comment says "re-run one stage"; its `convert` paragraph says "a failure stops there, keeps everything upstream, and `rerun` resumes". The resume reading reconciles both and is the only one that leaves a workdir artifact-consistent: every stage already clears or supersedes its downstream caches (survey and content unlink `monsters.json` and the area caches), so re-running one stage in isolation would strand the workdir mid-loop with fresh caches and stale artifacts, and the CLI would have to explain which commands to run next. One command, one consistent workdir. (Spec impact above.)
- Stage choices: `preprocess`, `survey`, `content`, `monsters`, `assemble` (geometry has no independent run — it lives inside assembly, phase 0's confirmed reading). Preconditions are the stage functions' own existing guards (`ValueError` on incomplete upstream). `provider` is required exactly when the resumed chain contains a model stage (`ValueError` otherwise names the stages that need it); `rerun(…, Stage.ASSEMBLE)` needs none and *is* the documented correction-loop `assemble`.
- **`rerun preprocess`** reads the workdir's own `source.pdf` (no PDF argument — the source is already the workdir's); `preprocess` learns to skip the self-copy when source and destination are the same file (`shutil.copyfile` raises `SameFileError` today — the one code change this requires). The fresh `RunMeta` it writes marks all stages pending, which is honest: everything downstream re-runs anyway. Phase 0's dangling pointer ("skip-if-unchanged logic belongs to `rerun`") resolves here, deliberately without change detection: the stage argument *is* the skip — the user names what to redo and everything upstream is kept verbatim; automatic staleness detection would need per-stage input hashing nothing else wants, for a decision the human has already made by choosing the stage.
- **Settings updates and the drift guard.** `settings_updates` (CLI: repeatable `--set KEY=VALUE`, values parsed with `yaml.safe_load` so `21`, `[21, 30]`, and `best-effort` all coerce naturally, then pydantic-validated against `ConversionSettings`) are applied to the `run.json` settings echo before the chain runs — the echo stays the single source of truth stages read. The guard that makes the phase 0 promise ("the settings echo … so `rerun` can detect settings drift") real: a pinned knob→owning-stage map (`render_dpi`/`max_pages`/`max_source_bytes`/`blank_page_renders` → preprocess; `survey_max_pages` → survey; `content_batch_pages` → content; `monster_fuzzy_threshold`/`monster_llm_top_k` → monsters; `unresolved_fallback` → assemble). Setting a knob whose owning stage is *upstream* of the rerun stage → `ValueError` telling the user which stage to rerun instead — the drifted echo would otherwise claim pages were rendered at a DPI they weren't. This is also how phase 2's deferral lands: flipping `unresolved_fallback` on an existing workdir is `rerun assemble --set unresolved_fallback=omit`, and the B3 content-filter workaround is `rerun preprocess --set 'blank_page_renders=[21]'`.
- `on_progress` events cover exactly the stages the resumed chain runs, in the phase 2 event shape — hosts stream a rerun the same way they stream a convert.
- Not in the façade: the spec's library API names `convert`, `assemble`, `check`, `estimate`, and `ConversionSettings`; `rerun` stays importable from `osrforge.convert` — one home per symbol, and the façade list is the spec's contract.

### 5. `estimate` — `estimate.py`

`estimate(pdf_path: Path, workdir: Path, settings: ConversionSettings | None = None) -> CostEstimate` — the spec's "preprocess only; rough token/cost estimate". It runs the real `preprocess()` into the given workdir (default `./<pdf-stem>.forge` at the CLI, mirroring `convert`): the licensing invariant forbids persisting module text outside the user's workdir, so a temp directory is not an option — and the workdir is warm for the human's next step (`rerun survey` continues from the rendered pages; a plain `convert` starts at preprocess by design and re-renders). Then pure arithmetic; no provider, no model call, no new dependency.

The heuristics, pinned from measured behavior (`docs/foundry-capabilities.md` and the four recorded full-module runs) and kept deliberately coarse — the spec asks for "rough", and the plan records the calibration so the error band is honest:

- Per-page tokens: text ≈ `ceil(text-layer chars / 4)` (the standing chars-per-token heuristic; JN1's measured survey implies ~880 text tokens/page against ~3.5 KB/page text layers, consistent) plus a flat **905 image tokens per page — DPI-independent**, the spike's measured constant (100/150/200 DPI cost identical tokens, so `render_dpi` never enters the formula).
- Survey input = page tokens for all pages + 2,000 (prompt + schema overhead); survey output = 70 × pages. Content input = **1.25 × page tokens** — pages are re-sent per level batch and map pages ride every batch, so a send-once model undercounts (JN1 measured 1.20×, minimod 0.77×, B3 ≈ 1.1×); content output = 550 × pages (the capability doc's dense-page constant). Monsters: flat 5,000 in / 500 out (JN1 measured 4,305/340; zero when a module resolves deterministically, unknowable in advance — the flat constant is the honest rough call).
- Pricing constants live here — `$2.50/1M` input, `$15.00/1M` output at the ≤ 272K tier — and `tools/extract/run_extraction.py` imports them (deleting its private copies; one home). The estimate applies the **272K single-request tier doubling** ($5.00/$22.50) to the survey request when its estimated input crosses the cliff — the one request that can.
- `CostEstimate` (frozen dataclass, the house result shape): page count, text/image token subtotals, per-stage estimated input/output, totals, USD, and `exceeds_survey_guard: bool` — when `page_count > survey_max_pages` the conversion would fail at survey before spending anything, and the estimate must say so rather than price a run that can't happen.
- Calibration, recorded here as the accuracy statement: against the four measured runs (JN1 186,744/31,763; The Hole in the Oak 139,272/16,089; B3 121,495/16,632; DCC #81 118,184/14,869) the formula must land within ±40% on input tokens — the amendment records the actual deltas, comparing like for like: all four measured totals predate the monsters stage, so the survey + content halves of the formula are compared against them and the flat monsters constant against JN1's separately measured 4,305/340. Schema retries and missing-key follow-ups are real tokens no pre-call estimate can see; the band, not the point value, is the contract.
- CLI `osrforge estimate <module.pdf> [--workdir DIR]` prints the table and the USD line hosts surface as their confirmation.

### 6. CLI polish and the façade — `cli.py`, `__init__.py`

- New subcommands wired to work items 3–5: `check`, `rerun <stage>` (choices exactly the five runnable stages; `--provider foundry`; repeatable `--set KEY=VALUE`), `estimate`. `convert` gains the same `--set` (first-run settings — a B3-like module needs `blank_page_renders` on its *first* conversion); one shared option-builder, one shared parser for `--set` values.
- `--version` prints `osrforge_version()` — table stakes for a tool whose artifacts embed the version.
- **The overrides template:** after a successful `convert`, if `overrides.yaml` does not exist, the *CLI handler* writes a comments-only template (the address grammar, one commented example per override kind, the reason rule) — it loads as the empty `Overrides` (`yaml.safe_load` of comments-only is `None`, already handled) and gives the human loop its on-ramp; never touched when the file exists. Library `convert()` does not write it (hosts own their correction UX and shouldn't get files they didn't ask for), and `assemble` does not either — overrides are input, and a pure function doesn't author its own inputs.
- The façade grows to the spec's complete promise: `check` and `estimate` join `convert`, `assemble`, `ConversionSettings`. Result types stay in their home modules.

### Implementation decisions the plan left open

- **`ResolutionMethod` grows `"override"`** (additive): the in-memory replaced resolution needed an honest method value; the `monsters.json` cache never contains it (the monsters stage alone writes the cache).
- **Town/module `null` semantics:** an explicit `null` clears the extracted value, returning the build to its extracted-empty path — the default and its `low_confidence:… unstated` flag apply exactly as for an extraction that came up empty. (Value replacement suppresses the flag, per the plan.)
- **A third `delve_incomplete` cause:** the party relocated mid-walk (a trap or transition fired). The walk's position assumption is void, nothing is provably wrong, and silence would hide the early end — the warning severity's "geometry up to that point played" reading covers it.
- **The goldens runner's two reports:** `goldens` always runs `check` and prints findings; the *written* `report.json` is the post-`check` report exactly when `--overrides` is given (the corrected gate's writer) and the post-assemble report otherwise (the uncorrected gate byte-compares assemble's own output, which emits `findings: ()` by design).
- **Golden re-bless for the additive `findings` field:** adding `findings: []` to every assembled report changed `report.json` bytes; the minimod and uncorrected JN1 goldens were regenerated by their documented flows in work item 1.
