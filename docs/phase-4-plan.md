# Phase 4 plan — measurement and release

Implementation plan for phase 4 of [the osr-forge spec](spec.md), the roadmap's final phase: eval corpus + scoring harness, docs, PyPI release — and survey chunking past the single-request guard, the one pipeline gap phases 1 through 3 each named phase 4 as the pickup for. The roadmap milestone, restated as this phase's bar: extraction quality is a measured number on a committed scoreboard instead of a vibe, and `osr-forge` 0.1.0 installs from PyPI with a live documentation site — the state osr-web's own phase 4 integration starts from. Every decision below rests on the measured behavior in [docs/foundry-capabilities.md](foundry-capabilities.md), the four recorded full-module runs, the committed JN1 and minimod asset sets, and osrlib-python's docs and release engineering — the workflow-parity reference `AGENTS.md` names.

Four facts shape everything and are worth stating before the work items:

- **Prompt and schema bytes are load-bearing.** Every committed fixture's request digest hashes the system prompt, the schema, and the page bytes; any edit to `SURVEY_SYSTEM`, `SURVEY_SCHEMA`, or the single-request build path strands the minimod and chaotic-caves fixture sets (the standing re-record rule). Survey chunking must therefore be purely additive: a source at or under `survey_max_pages` pages must produce byte-identical requests to today's, and the committed replay gates are the proof.
- **Extraction is nondeterministic run to run.** The same module through the same prompts yields different token counts and different marginal extractions on every live run — there is no seed on the chat surface. So the scoreboard records *specific runs*, regression detection compares against a noise band measured from repeated baseline sweeps rather than asserting equality, and evals stay out of CI exactly as the spec demands (on-demand harness, never per-commit).
- **The licensing fence binds the repo and the dist artifacts, not users.** The corpus ships pointers + hashes, never PDFs — but the hand-checked ground truth is *derived text* (area keys, monster names, treasure letters), so corpus modules must be freely licensed under phase 0's verification procedure (the module PDF's own license page states CC BY-SA 4.0 and its contributors appear on the project's consent list) before one line of truth is committed. On the other side of the fence, the release audit must prove the wheel and sdist carry no `tests/` or `tools/` content at all — no game text, no fixtures, no PDFs.
- **The docs toolchain has a known supply-chain hazard.** osrlib's pins are the evidence: `mkdocs-gen-files` and `mkdocs-literate-nav` next releases (0.6.1 and 0.6.3 respectively) depend on `properdocs`, an unvetted mkdocs fork that ships an import hook redirecting all `mkdocs.*` imports to itself. osr-forge pins both exactly (`mkdocs-gen-files==0.5.0`, `mkdocs-literate-nav==0.6.1`) and carries the do-not-bump-without-inspection warning verbatim.

## Scope

In scope:

- Survey chunking past the guard — the phase 1/2/3 deferral: chunked survey requests over page windows with a deterministic raw-level merge, `estimate` reshaped to price chunked surveys (and `exceeds_survey_guard` removed — the guard failure it reported no longer exists)
- The eval corpus — `tools/eval/corpus/`: per-module manifests (pointer + sha256 + license verification note) and hand-checked structural ground truth; v1 members minimod, JN1 The Chaotic Caves, and one held-out license-verified BFRPG module
- The scoring harness — `tools/eval/run_eval.py`: pinned metrics over the stage caches, the committed scoreboard, the double-run baseline sweep that sets the noise band, and the regression rule; scorer unit tests plus the JN1 pinned-baseline test in CI with zero network
- The docs site — mkdocs-material + mkdocstrings in osrlib parity: generated API reference and vocabulary tables, hand-written guides, strict CI build, Pages deploy from `main`
- Release engineering — version 0.1.0, `CHANGELOG.md`, pyproject metadata, `tools/release/` (dist audit + install smoke), the tag-driven `release.yml`, the releasing runbook in `AGENTS.md`, the artifact-contract freeze, and the v0.1.0 release itself after the implementation PR merges
- Tests for all of the above, green in CI with no network

Out of scope: the roadmap ends with this phase, so deferrals land in the spec's open-questions register and future releases plan their own work rather than a numbered phase picking them up. Vision-first geometry (the corpus built here is exactly what the spec says is needed to judge it — the named next candidate); custom `MonsterTemplate` emission; non-PDF inputs; OCR fallback; additional providers beyond Foundry (the spec's "drop-ins later"); full automated playthroughs; versioned documentation (osrlib's adoption trigger carried verbatim: the first post-1.0 release whose published docs must describe behavior different from `main` adopts mike in that release's own plan); per-commit CI evals (the spec forbids them — the harness stays on-demand); corpus growth past the v1 members (standing and additive — a new module is a manifest, a truth file, and a sweep, no code); auto-download in the eval harness (manifests carry source URLs but fetching stays manual — the sha256 check is the integrity gate, URLs rot, and the harness takes no HTTP dependency).

**Constraint carried from phases 2 and 3, with one carve-out: no edit to existing prompt, schema, or alias-table bytes.** The chunked survey path adds a chunk preamble to its *own* requests only; the single-request path — every source at or under the guard — builds byte-identical requests, and the committed minimod and JN1 replay gates prove it on every CI run. `MONSTER_ALIASES` stays frozen.

Spec impacts, applied with the implementation PR so the spec stays the authority:

- **§ Pipeline (survey):** "chunked if needed" gets its one-sentence definition: sources over the survey chunk size are surveyed in page windows of that size, each request carrying its window's pages plus a preamble naming the window, and the windows' answers are merged deterministically before normalization.
- **§ Configuration:** the knob list gains the survey chunk size (`survey_max_pages` — the phase 1 guard's meaning changes from "larger sources fail" to "larger sources chunk").
- **§ Testing and evals:** the eval bullet is pinned to what ships: the four metric families by name, the scoreboard's home, the double-run noise band, the regression rule (which changes re-run the sweep, and what a metric drop requires), and the ground-truth authoring discipline (from the printed module, never from pipeline output).
- **§ Tooling and packaging:** gains the Documentation URL (`https://mmacy.github.io/osr-forge/`) and the released-version statement (0.1.0, Development Status Beta).
- **`AGENTS.md`:** gains a Releasing section (osrlib-parity runbook: changelog discipline, annotated tag, local dry run, what `release.yml` does); the artifact-contract freeze trigger tightens from "once osr-web integrates" to "from the first PyPI release" (an installable package *is* external consumption — waiting for a named consumer would leave the freeze moment unobservable); and the eval regression rule lands beside the fixture re-record rule it always accompanies (the same edits trigger both).
- **`README.md`:** status section rewritten for release (install from PyPI, docs link), and the `survey_max_pages` knob row updated to the chunk-size meaning.

## Work items

### 1. Survey chunking — `survey.py`, `estimate.py`

The deferral, closed where phases 1–3 said it would be. Today `survey()` raises `ExtractionError` when `page_count > survey_max_pages`; the guard becomes the chunk size and the error disappears.

- **Windows.** Pages split into contiguous, disjoint windows of `survey_max_pages` pages: `1..K`, `K+1..2K`, …. No overlap, pinned: overlapping windows would double-extract boundary areas and force the merge to arbitrate conflicting duplicates of the same key, and the failure overlap would guard against — a dungeon spanning a boundary — is already covered because each window reports the parts it saw and the merge unions them. Page markers stay absolute (`page_request_parts` takes the window's range; `[page N]` markers and `source_pages` already speak absolute numbers), so downstream stages need no change at all.
- **Requests.** Each window's request is `SURVEY_SCHEMA` unchanged plus `SURVEY_SYSTEM` with an appended chunk preamble ("this request carries pages N–M of a P-page module; other pages arrive in separate requests; survey only what these pages show"). The preamble is appended *only* in chunked mode — the single-request path calls today's `build_survey_request` untouched, which is what keeps every committed fixture valid. Tag stays `survey`; distinct windows produce distinct request digests, so fixtures and caching need nothing new. `track_stage` already accumulates usage across multiple `add_usage` calls.
- **Merge, at the raw level, before one `normalize_survey` pass.** Merging *normalized* indexes would fight reserve-then-bump uniquing (a key bumped to `5-2` in one window and not the other could never re-join), so the merge joins the windows' schema-valid raw answers and the merged raw dict flows through the existing `normalize_survey` exactly once. Join rules, all deterministic and first-seen-order preserving: dungeons join on `canonical_slug(name)`; within a joined dungeon, levels join on `number`; within a joined level, areas join on `canonical_slug(key)` — first occurrence wins every scalar field, `source_pages` and `map_pages` union. `title` and `town` take the first non-empty occurrence in window order (`town` as a unit: first entry with a non-empty name, else first with a non-empty description, else empty); `hooks` concatenate deduplicated by exact string; `monster_names` union in first-seen order. The merge function is public and pure (`merge_survey_answers`), unit-testable without a provider.
- **Honesty about the seam.** A window that sees only a dungeon's back half can misattribute or thinly describe areas whose context lived in the other window; the merge unions what each window actually saw and the correction loop remains the backstop. No committed module chunks (every asset is ≤ 48 pages against a 150-page chunk size), so chunked behavior is exercised by scripted-stub tests, not recorded fixtures — and that is fine, because the merge is deterministic code and the model-facing halves (schema, page parts) are the proven single-request shapes.
- **`estimate` reshape.** `CostEstimate.exceeds_survey_guard` is removed outright — greenfield discipline, no deprecation residue; the run it warned could not happen now happens. When `pages > survey_max_pages`, the survey estimate becomes per-window: each window prices its own pages' tokens plus the flat 2,000 prompt/schema overhead, and the 272K tier-doubling branch no longer applies to chunked surveys (each window is at most the chunk size, ≈ 240K tokens at the default — under the cliff by construction; the branch keeps applying to the single-request case, the one request that can still cross when the knob is raised). CLI table and README row updated; pinned expected numbers in tests.

### 2. The eval corpus — `tools/eval/corpus/`

A corpus member is a directory, `tools/eval/corpus/<module-id>/`, holding exactly two files. The corpus lives under `tools/` beside `spike/` and `extract/` — repo-only, never packaged, live-network only through the harness that consumes it.

- **`manifest.yaml`:** `title`, `source_url` (the pointer — where a human downloads the PDF), `sha256` (of the exact PDF the truth was authored against; the harness refuses a mismatched file before any spend), `pages`, and `license` (SPDX id plus a `verified` note recording how and when the phase 0 procedure was run: the PDF's own license page states CC BY-SA 4.0 and the contributors appear on the BFRPG consent list). The manifest is the whole redistribution surface — no PDF is ever committed, honoring the spec's pointer + hash rule.
- **`truth.yaml`:** hand-checked structural ground truth, deliberately prose-free to keep the licensing surface thin — printed keys, names, and codes only, riding a CC BY-SA attribution notice in `tools/eval/README.md`. Per dungeon (printed `name`), per level (`number`), per area (printed `key`): `encounters` (printed monster `name`; `template` — the osrlib catalog id the name should resolve to, omitted when the module's monster genuinely has no SRD template; `count`, omitted when the module states none or a variable one), `connections` (printed keys of same-level areas the module says connect), and `treasure` (`present`, plus `letters` only when the module states treasure-type letter codes).
- **Authoring discipline, pinned:** truth is authored from the printed module, never from pipeline output — reading `survey.json` before writing a module's truth file contaminates the measurement with the thing being measured. The phase 3 correction session's notes are legitimate references for JN1 because that session verified every claim against the printed pages. The amendment records, per module, who checked what against which printing.
- **v1 members, pinned:** `minimod` (CC0, authored in-repo — its manifest points at the committed PDF and generator; trivially truth-checked, and the zero-cost smoke of the whole harness), `jn1-chaotic-caves` (CC BY-SA 4.0, license-verified in phase 0; pointer to the BFRPG showcase, the already-recorded sha256), and `bf1-morgansfort` (BF1 Morgansfort, the named candidate, *subject to the same license verification before anything is committed*; JN2 The Monkey Isle and JN3 Sakatha's Tomb are the fallback candidates if it fails). The third member is the corpus's whole point: **minimod and JN1 tuned every prompt in the pipeline, so they measure regression, not quality — BF1 is held out, the first module the prompts were never developed against, and the only honest quality number in v1.** Recorded as a standing limitation: all v1 members share BFRPG layout conventions; corpus diversity is future, additive growth.

### 3. The scoring harness — `tools/eval/run_eval.py`

Manual and repo-only like the extraction runner; scoring itself is pure code over a workdir's stage caches — offline, deterministic, and unit-tested in CI.

- **What the scorer reads: the stage caches, never `adventure.json`.** Evals measure *extraction*; assembly's best-effort fallbacks exist to mask extraction gaps in the playable draft, which is exactly what a measurement must not let them do. Area recall reads `survey.json`; encounters and connections read the `areas.*.json` content caches; resolution reads `monsters.json`; treasure reads the content caches' raw strings through `parse_treasure` imported from `osrforge.assemble` — the one grammar, one home (the estimate-pricing-constants precedent; a private reimplementation would drift).
- **Alignment, pinned.** Truth dungeons match extracted dungeons greedily by area-key-set overlap (both sides' keys through `canonical_slug`), ties broken by `difflib` ratio over name slugs, each dungeon matched at most once; levels align by number within a matched dungeon; areas match on slug equality within an aligned level. An unmatched truth dungeon counts all its areas as misses — a survey that missed a whole lair scores like it.
- **The metric families, pinned** (each reported per module and as the corpus mean):
    - *Areas:* recall (matched truth areas / truth areas) — the spec's named metric — plus precision (matched extracted / extracted) as the hallucination guard.
    - *Encounters:* name recall (truth encounters whose printed name, under `normalize_monster_name` on both sides, appears in the matched area's extracted encounters), count accuracy (name-matched encounters whose counts agree, over those where truth states a count), and resolution accuracy (name-matched encounters whose `monsters.json` resolution equals truth's `template`, over those where truth names one; truth entries without a `template` are excluded from the denominator and reported as a `non_srd` tally).
    - *Connections:* F1 over undirected same-level edges whose endpoints are both matched areas, extracted connections vs truth's.
    - *Treasure:* presence agreement (matched areas where extracted-treasure-signal equals truth's `present`) and letter accuracy (areas where truth states letters and the parsed letters match exactly).
- **CLI shape:** `run_eval.py convert <module-id> <pdf-path> [--workdir DIR]` — verifies the PDF's sha256 against the manifest *before any model spend*, then runs the package `convert` with `FoundryProvider` (same `OSRFORGE_FOUNDRY_*` env vars as the extraction runner; recording off — evals produce scores, not fixtures); `run_eval.py score <module-id> --workdir DIR [--update-scoreboard]` — offline scoring, printing the metric table; `run_eval.py report` — renders the committed scoreboard.
- **The scoreboard:** `tools/eval/scoreboard.json`, committed. Per module: `run` metadata (date, `model_id`, `osrforge_version`, input/output tokens, USD at the pinned pricing constants) and the `metrics` block. Run metadata is injectable into the scoring function so the CI determinism test can pin bytes.
- **The baseline sweep and the noise band.** The implementation runs the full sweep **twice** (≈ $3 per sweep at the measured rates — JN1 ≈ $0.95, BF1 at ~84 pages ≈ $1.60, minimod ≈ cents); the second run exists to *measure* run-to-run noise instead of guessing it. The committed scoreboard carries the first run; the amendment records both runs' metrics side by side, and the regression band is set to the observed per-metric spread, floored at 0.02 absolute.
- **The regression rule** (recorded in `AGENTS.md` beside the fixture re-record rule, and in the spec): any PR that edits extraction prompts or schemas, `MONSTER_ALIASES`, resolution logic, or the model deployment re-runs the sweep and commits the updated scoreboard in the same PR — the same edits that strand fixtures re-measure quality, one workflow, two obligations. A metric dropping by more than the band requires an explicit justification in the PR description; silence is a blocked merge.
- **Scorer tests, CI, zero network:** synthetic truth-plus-caches tables driving every metric family through empty, perfect, and partial cases; alignment edge cases (missed dungeon, bumped keys, tie-broken names); determinism (score twice, byte-identical output with pinned run metadata); and the JN1 pinned baseline — fabricate the workdir from the committed chaotic-caves caches (the existing goldens-test pattern), score against the committed JN1 truth, assert the exact numbers. Those numbers re-bless only when the caches or the truth file deliberately change, which makes the scorer's behavior itself a golden.

### 4. The docs site — `mkdocs.yml`, `docs/`, `tools/docs/`

mkdocs-material + mkdocstrings in deliberate parity with osrlib-python's setup — same theme configuration, same markdown extensions (`attr_list`, `pymdownx.highlight`/`inlinehilite`/`superfences`, `snippets` with `check_paths: true`), same `strict: true` and validation block, same `watch: [src]`, same Pages-artifact deployment with no `gh-pages` branch. The docstrings have followed the mkdocstrings-compatible house style from birth, so this work item is site assembly, not a docstring retrofit; the strict build with `autorefs` is what surfaces any broken cross-reference, and fixing what it flags is in scope.

- **`site_url`:** `https://mmacy.github.io/osr-forge/`. **`exclude_docs`:** `spec.md`, `phase-*-plan.md`, `foundry-capabilities.md` — design documents stay in `docs/` beside the site pages, unpublished, exactly osrlib's pattern.
- **Dependency group:** `docs` group included by `dev` (one `uv sync`), with the exact `mkdocs-gen-files==0.5.0` / `mkdocs-literate-nav==0.6.1` pins and the properdocs warning comment carried from osrlib.
- **Generated pages** (`tools/docs/`, gen-files scripts): the API reference over `src/osrforge` via mkdocstrings + literate-nav (osrlib's `gen_api_reference` pattern), and a vocabulary page generated from the code — the `Flag` enum, the `LintCheck` enum with the producer's id→severity table — so the badge vocabularies UIs depend on can never drift from their one source. `inventories`: python, pydantic, and osrlib's own `objects.inv` (`https://mmacy.github.io/osrlib-python/objects.inv`) — osr-forge docstrings reference osrlib symbols constantly, and they should resolve to osrlib's published reference (verify the inventory URL serves at implementation time; mkdocstrings sites publish it by default).
- **Hand-written pages, with one-line charters:** Home (what osr-forge is, the four artifacts, front-end-agnostic by contract); Getting started — install (uv/pip, the `entra` extra) and a first conversion (`estimate` → `convert` → artifact tour); Guides — the correction loop (the README walkthrough, expanded to the full report/previews/overrides/`assemble && check` cycle), settings and `rerun --set` (the knob table and the drift guard), provider setup (Foundry env vars, key vs Entra auth); Reference — CLI (per-subcommand, hand-written; the argparse surface is small and stable), the workdir and artifacts (`run.json`, `report.json` fields, `overrides.yaml` kinds and the three application rules, addressed with the generated vocabulary page); Evals (the corpus rules, running the harness, reading the scoreboard); Licensing (MIT, no game content in the package, the fence and — verbatim from `AGENTS.md` — what it does and doesn't bind).
- **CI:** a `docs` job on every push/PR (`uv run mkdocs build --strict`, ubuntu only — platform-independent), uploading the Pages artifact and deploying on `main` pushes via `actions/deploy-pages` — osrlib's `ci.yml` jobs, mirrored. One-time repository setup, named for the human: set the Pages source to "GitHub Actions".

### 5. Release engineering — `pyproject.toml`, `CHANGELOG.md`, `tools/release/`, `release.yml`

- **Version 0.1.0, pinned, with the freeze it triggers.** The spec's own report example stamps `0.1.0`; a 1.0 claim waits for what this phase creates the means to earn — eval-driven prompt stability and osr-web integration feedback. Classifier `Development Status :: 4 - Beta`. From this release, the artifact contracts (`adventure.json` stamped document, `report.json` flag and finding vocabularies under `schema_version`, the `overrides.yaml` schema) are additive-only within a schema version — `AGENTS.md`'s freeze wording updates from "once osr-web integrates" to "from the first PyPI release". `SCHEMA_VERSION` stays 1; nothing about releasing changes artifact semantics.
- **pyproject metadata:** `keywords` (bx, osr, ttrpg, adventure-module, pdf-conversion, llm, osrlib, …), classifiers (Development Status, Intended Audience :: Developers, OS Independent, Python :: 3 / 3.14 only — the version tested in CI, osrlib's rule; `Typing :: Typed`; no `License ::` classifier beside the SPDX expression, per PEP 639), and URLs: `Documentation` (the Pages site), `Issues`, `Changelog`.
- **`CHANGELOG.md`:** Keep a Changelog format, osrlib parity. The initial `0.1.0` section summarizes the shipped surface by phase (pipeline, correction loop, evals, docs) rather than replaying history; the `[Unreleased]` discipline — a PR that changes user-visible behavior adds its bullet in the same PR — enters `AGENTS.md` with the Releasing section.
- **`tools/release/check_dist.py`:** audits both dist artifacts against the tag version: wheel contains `osrforge/**` (with `py.typed`) and its dist-info only; sdist contains the source layout, `pyproject.toml`, `README.md`, `LICENSE`, and lockfile-adjacent files only; **neither artifact contains any path under `tests/` or `tools/`, any `.pdf`, `.png`, or fixture JSON** — the licensing fence, machine-checked at the release gate. `tools/release/install_smoke.py`: into a fresh venv from the built wheel — import `osrforge`, `metadata.version` matches the tag, the façade exports exactly the spec's library API (`convert`, `assemble`, `check`, `estimate`, `ConversionSettings`), `osrforge --version` prints the version, and `import osrlib` resolves through the pinned compatible range.
- **`release.yml`:** osrlib's tag-driven shape, mirrored minus the SRD-data job (osr-forge has no generated-data drift to gate): `check-tag` (tag matches pyproject version) → `test` matrix + `docs` build (duplicated from `ci.yml` by hand, with the keep-in-sync comment — a tag push triggers no CI run) → `build` once + `check_dist` audit → `install-smoke` on both OSes from the exact built artifact → `publish` via OIDC trusted publishing (environment `pypi`, no credentials in the repo) → GitHub Release from the tagged version's changelog section, with the same has-content guard.
- **The runbook** (`AGENTS.md` Releasing section, osrlib parity): changelog discipline; release = annotated `vX.Y.Z` tag on the merge commit; the local dry run before tagging (`uv build`, `check_dist`, fresh-venv `install_smoke`); versioned-docs adoption trigger carried verbatim.
- **One-time manual setup, named for the human (account-level, not CI-automatable):** a PyPI pending publisher for project `osr-forge` (workflow `release.yml`, environment `pypi`), the `pypi` environment in the GitHub repo, and the Pages source flip from work item 4.
- **The release act.** After the implementation PR merges: run the local dry run, tag `v0.1.0`, let `release.yml` publish, verify `uv pip install osr-forge` from PyPI and the live docs. The amendment records the release evidence (PyPI URL, docs URL, workflow run) — the phase's milestone is not "ready to release" but *released*.

### 6. Tests

- **Chunking:** the merge-function table (boundary-split dungeon re-joining on slug, conflicting scalar fields resolving first-seen, page-list unions, town/title first-non-empty, hooks dedup, monster-name union order); a scripted-stub chunked run over a synthetic multi-page workdir with `survey_max_pages` forced small (windows built correctly, absolute page markers, one normalize pass, usage accumulated, cache cleared once on success); an explicit fingerprint-stability assertion that the at-or-under-guard path builds a request digest-identical to the committed minimod survey fixture's (the replay gates already prove this end to end; the unit test names the invariant so a future edit fails with a message, not a mysterious fixture miss).
- **Estimate:** per-window pricing at pinned numbers for a synthetic over-guard page count; the tier-doubling branch still firing for a single-request survey with the knob raised; `exceeds_survey_guard` gone from the result type and the CLI table.
- **Scorer:** the work item 3 suite — synthetic metric tables, alignment edge cases, determinism, and the JN1 pinned baseline over committed caches and truth.
- **Harness plumbing:** manifest loading and sha256 refusal (a doctored byte fails before any provider construction); scoreboard read/update round-trip with injected run metadata.
- **Docs:** `mkdocs build --strict` in CI is the gate (broken nav, snippet, or cross-reference fails the build); the gen scripts run inside it.
- **Release tooling:** exercised by the documented local dry run and the release workflow itself, osrlib parity — `check_dist` and `install_smoke` are gates, not test-suite members; CI does not build dists per-commit.
- All green under `uv run pytest` locally and in CI on both OSes, no network anywhere.

## Sequencing

1. Survey chunking and the `estimate` reshape (work item 1) — code first, so the baseline sweep measures the pipeline as it will ship.
2. The corpus (work item 2) — BF1 license verification before anything else lands in its directory; manifests and truth authoring for all three members under the printed-module discipline.
3. The scorer and its tests (work item 3, code half) — metrics against synthetic tables, then the JN1 pinned baseline over the committed caches.
4. The baseline sweep, twice, and the committed scoreboard plus the measured noise band (work item 3, measurement half) — after 1–3, before docs quote any numbers.
5. The docs site (work item 4) — content references the finished harness and scoreboard.
6. Release engineering (work item 5) — last among code; the changelog and metadata describe what actually shipped.
7. README, spec, and `AGENTS.md` impacts, plus a final pass tracing every phase 4 roadmap contract to code and tests or to a named entry in the open-questions register.
8. Post-merge: the one-time PyPI/Pages setup confirmed, the local dry run, the `v0.1.0` tag, publish verified, amendment recorded.
