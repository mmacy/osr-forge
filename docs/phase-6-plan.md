# Phase 6 plan — playable structure

Implementation plan for phase 6 of [the osr-forge spec](spec.md): doors, stairs, and the other structural signal the pipeline reads but cannot yet say. The first consumer's evidence is unambiguous — across every real conversion in osr-web's adventure library (B3, B4, DCC #81, *The Hole in the Oak*), 100% of emitted edges are `open`, so the consumer's entire door subsystem (open, force, listen, pick, secret-door rendering) is dead code against forge output; B3's level 2 (28 areas) has no transitions at all; DCC #81's levels 2–4 are unreachable from the entrance; and `Adventure.description` and `town.services` are empty in every document. The milestone: **a multi-level module converts to a draft whose printed structure actually plays — every level reachable through real stairs, doors that are doors (secret ones included), module metadata populated — with the improvement measured on the committed scoreboard and the BYOM baseline, not vibed.** The roadmap entry for this phase is a spec impact applied with the implementation PR, as phase 5's was.

Four facts shape everything:

- **The output format is not the constraint, and neither is the renderer.** osrlib 1.1.0 already expresses everything this phase emits — `Edge`/`DoorSpec` (normal and secret, stuck, locked), `TransitionSpec`, `TownSpec.services`, `Adventure.description` — and the preview renderer has drawn door ticks and transition glyphs since phase 2, built deliberately ahead of synthesis for the override path. No osrlib change is needed anywhere in this phase; the one conversion capability that *does* need an osrlib seam (custom monster emission) is scoped out below with its prerequisite named.
- **The model already reports the missing structure; the schema drops it on the floor.** B3's stage cache carries area 48's vertical connection to `"Second Level"`, and DCC #81's carries `"rock passage"`, `"secret entrance in nearby passage"` — level-shaped and mechanism-shaped answers that `AreaConnection`'s area-keyed `to_key` cannot resolve, so they fall out as `connection_ambiguous` and whole levels orphan. Doors are the same story told in reverse: the spec pinned "door placement is override territory in v1 — the extraction schema carries no door signal," so prose like "a stuck door in the north wall" extracts as a bare direction. Phase 6 is mostly about widening one small schema so the answers the model is already giving have somewhere to land.
- **Phase 5 built the baseline this phase is judged against — and named its customers.** Prompts and schemas have been frozen since phase 1; the phase 5 amendment records the survey mode-flip (1 in 4 known JN1 runs) and the treasure grammar's two failing shapes (B3 carries 37 `treasure_unparsed` flags, named magic items among them) as the two candidates "now judgeable against this BYOM baseline." This phase is that judgment: every prompt, schema, and grammar change here re-runs the sweep under the standing regression rule, and the BYOM four re-convert and re-score against the same truth files.
- **The yardstick has a recorded crack, and it gets fixed before anything is measured through it.** Phase 5's B4 finding: the scorer aligns levels by number, so a legitimate level-modeling disagreement (truth asserts the module's 10 tiers; extraction grouped the same 117 keys into the 6 printed dungeon levels) scored 0.0171 area recall against a draft that had found everything. Survey-discipline changes land in exactly that blast radius, so the alignment fix is this phase's first work item, sequenced before any prompt is touched.

## Scope

In scope:

- Connection structure in the content schema — `via` (door, secret door, stairs, trapdoor, chute), door state, and level-targeted vertical links — with the prompt changes to populate it
- Survey additions — the module's own description, town services — and survey site discipline: the phantom-dungeon rule (B4's five single-area letter-feature "dungeons") and the multi-lair counting rule (the mode-flip)
- Geometry synthesis of door and secret-door edges and of transitions from level-targeted links, with a pinned landing policy and its report flag
- Assembly: metadata fill, the two treasure-grammar extensions, and the flag-vocabulary addition
- The scorer's level alignment, rebuilt on area-key overlap (the B4 fix), with the JN1 pinned baseline re-blessed
- The full measurement pass the standing rules require: fixture re-records, a double corpus sweep with a refreshed noise band, and BYOM re-runs — all live work owner-executed per the phase 5 permission note

Out of scope, each with its disposition:

- **Custom monster emission** — the fix for DCC #81's measured 0.0 resolution accuracy (every encounter a flagged SRD stand-in) — is the named phase 7 candidate, and its prerequisite is now verified rather than assumed: osrlib has no custom-catalog seam today (`GameSession.new` hardcodes `load_monsters()`/`load_equipment()`, and roughly ten engine call sites reach for the module-level loaders directly), so bundled `MonsterTemplate`s are an osrlib change first — additive, semver-minor, run through osrlib's own two-PR phase loop — and an osr-forge stat-block extraction phase second. The spec's open question is rewritten to carry this finding (spec impact below).
- **Door and transition eval metric families** — deferred. Measuring door recall or transition recall requires truth-format assertions (`via` on truth connections, cross-level link assertions) and re-authoring across five-plus truth files; this phase measures structure through the playability lint (`area_unreachable` counts on multi-level drafts collapse when transitions synthesize) and the existing connections family, and records the truth-format cost so a future phase can pick it up deliberately.
- **Vision-first geometry** — unchanged open question. This phase's door and stair signal comes from the key's prose, not the map image; a map-vision pass remains the future candidate for geometry that matches the printed cartography.
- **Party and pregen extraction** — new open question, recorded from consumer evidence (osr-web hardcodes a six-member roster and per-class equipment kits because `adventure.json` has no party surface; osrlib already has `party_to_document`). Not this phase.
- **`travel_turns` extraction** — stays empty. Modules rarely state town-to-dungeon travel in a convertible form, and the first consumer gates town commands on mode, not the field; inventing a number would violate the no-custom-content posture.
- **The survey mode-flip as a solved problem** — this phase changes the prompt and measures; it does not promise the flip's elimination. The acceptance bar is pinned in work item 2.
- **OCR fallback and non-PDF inputs** — spec open questions, unchanged.

Spec impacts, applied with the implementation PR:

- **§ Roadmap** gains entry 7: *Phase 6 — playable structure.* Door and stair extraction and synthesis, survey site discipline, module metadata, treasure-grammar extensions, and overlap-based eval level alignment. Milestone: a multi-level module converts to a draft whose printed structure plays — every level reachable, doors that are doors — measured against the phase 5 baselines.
- **§ Pipeline stage 3 (content)** adds the connection structure to the extracted-facts list; **stage 5 (geometry)** replaces "emit open `Edge`s (door placement is override territory in v1 — the extraction schema carries no door signal; a map-vision pass is the future automatic candidate)" with the door/transition synthesis described here, overrides remaining the last word.
- **§ Extraction report**: the flag vocabulary sentence gains `transition guessed`.
- **§ Open questions**: the custom-monster entry is rewritten to record the verified osrlib seam gap and the two-repo sequencing; a party/pregen extraction entry is added; the vision-first geometry entry is narrowed to map-image geometry (prose-signal doors and stairs are no longer part of the question).
- **`README.md` and the docs site**: the pipeline description's geometry sentence, the report reference's flag table (generated badge vocabulary regenerates), and the correction-loop guide's door example gains a "when extraction already placed the door" caveat.

## Work items

### 1. Scorer level alignment by area-key overlap — `evals.py`

- Within an aligned dungeon, truth levels and extracted levels align by **maximal area-key overlap** instead of by number: compute the overlap matrix (canonical-slug key intersection counts), take pairs greedily by descending overlap, break ties by smaller level-number distance and then by lower truth number, and leave zero-overlap levels unmatched. Area alignment within matched level pairs is unchanged, as is dungeon alignment — B4's disagreement lives entirely at the level tier, and the dungeon-count fields phase 5 added stay as they are.
- The committed corpus is unaffected by construction (minimod's two levels and JN1's fourteen single-level lairs align identically under both rules — a repo test pins this equivalence on the committed caches), so the JN1 pinned baseline re-blesses with unchanged numbers and the noise band carries until this phase's sweep refreshes it. B4 heals in the private board: each truth tier's keys sit inside exactly one extracted coarse level, so the 117 keys align and the 0.0171 artifact resolves into a real measurement. Sequenced first: the survey-discipline work below moves level groupings by design, and it must be measured through the fixed yardstick, not the cracked one.
- Re-scoring the private BYOM board is offline and free (`score --corpus ~/Documents/osr-forge-byom --workdir …`); the refreshed numbers land in this phase's amendment. Publication remains deferred on the owner-sampling leg exactly as phase 5 left it — nothing here changes that posture.

### 2. Survey: metadata and site discipline — `survey.py`, `contracts/stages.py`

- **Schema additions, both additive to the survey cache.** `SurveyIndex` gains `description: str` — the module's own pitch, condensed from its printed introduction or back-cover text, empty when the module has none (extraction of existing prose, exactly like `hooks`; never composition). `TownInfo` gains `services: tuple[str, ...]` — the named establishments and services the module states (an inn, a temple, a general store). Assembly maps them onto `Adventure.description` and `TownSpec.services`, both today's hardwired defaults.
- **The phantom-dungeon rule.** B4's survey coined five single-area dungeons from unkeyed letter features on the underground-city map (b/k/o/q/r) — each became a stub "dungeon" with an Enter button in osr-web. The prompt's dungeon definition gains the rule: a dungeon exists only where the module prints a keyed area list for it; lettered or unkeyed map callouts inside another site's map are that site's features, not sites. Acceptance: a B4 re-conversion surveys one dungeon (the extracted-dungeons count phase 5 added makes this legible in the scoreboard entry).
- **The multi-lair counting rule.** The mode-flip's shape is known (JN1's ten lairs collapsing into one dungeon, 1 run in 4): the existing one-dungeon-per-keyed-site instruction gets a counting-anchored restatement — enumerate the keyed sites first, then emit exactly that many dungeons. Wording is implementer latitude; the acceptance bar is not: JN1 surveys 14/14/14 dungeons in both sweep runs of this phase, and any flip that still occurs is recorded in the amendment with the run kept, per the no-cherry-picking rule.
- Both are frozen-prompt changes — the whole point of phase 5's baseline. The eval regression rule fires (work item 6), and every recorded fixture whose request embeds the survey prompt or schema strands and re-records.

### 3. Connection structure in the content schema — `contracts/stages.py`, `content.py`

`AreaConnection` widens from `(to_key, direction)` to carry mechanism and level-shaped targets — every field additive to the content cache, every one an answer the model is already trying to give:

- `via: Literal["passage", "door", "secret_door", "stairs", "trapdoor", "chute", "other"] = "passage"` — the stated mechanism, `passage` when the text names none. `secret_door` is its own value, not a modifier: it drives both `DoorSpec(kind="secret")` and the playability lint's `secret_only_access` warning, which becomes meaningful for the first time.
- `door_stuck: bool = False`, `door_locked: bool = False` — stated door conditions; a model validator rejects them on non-door `via` values (defense in depth behind the extraction schema, mirroring the `count_dice` posture).
- `to_key: str | None` and `to_level: int | None`, at least one required — the escape hatch for exactly the B3 failure: "stairs descend to the second level" states a level, not a keyed area. The prompt instructs: prefer the keyed target; else state the level; the free-string-in-`to_key` fallback stays for anything else and keeps flowing to `connection_ambiguous` as today.
- Prompt guidance names the signals: doors, secret doors, stairs, trapdoors, chutes, stuck/locked conditions, and the level-vs-area target preference. Same-level resolution, batching, and the direction rules are untouched.

### 4. Geometry: doors and transitions — `geometry.py`, `assemble.py`

- **Door edges.** When a placed same-level connection carries `via` of `door` or `secret_door`, the edge where the route leaves the source area's boundary becomes `Edge(kind="door", door=DoorSpec(kind=…, stuck=…, locked=…))` — the prose describes the door in the wall of the room being described, so the source end is the faithful placement; for a direct adjacency there is only one shared edge and the question is moot. Duplicate mentions keep the first stated door facts in survey order, consistent with the existing first-mention rules. Overrides remain the last word: the geometry `edges` override already expresses every door correction, unchanged.
- **Transitions from level-targeted links.** A vertical connection with `to_level` (and no resolvable `to_key`) becomes a `_CrossLevelLink` whose target area is guessed deterministically: the target level's first keyed area in survey order, unless that level carries a reciprocal vertical mention pointing back at the source level, which pairs instead. The transition kind derives from `via` (`stairs` → `stairs_up`/`stairs_down` by direction, `trapdoor` and `chute` as themselves — both one-way by design, already exempt from `transition_unpaired`); a vertical link whose `via` is anything else defaults to stairs, the overwhelmingly common printed mechanism. Reciprocal emission, cell assignment, and the existing keyed-target path are untouched.
- **The flag.** Every guessed landing flags the source area `transition_guessed:<dungeon>/<level>/<area>` (the guessed landing's address) — a new `Flag` member, additive within `report.json`'s `schema_version` per the artifact contract, and the thing a review UI badges so a human confirms the landing or corrects it through the existing geometry `transitions` override. The spec's flag-vocabulary sentence and the generated docs badge table update with it.
- **No preview or lint work.** The SVG renderer has drawn door ticks (outlined when secret) and transition glyphs since phase 2; `secret_only_access` and `transition_unpaired` were specified for exactly this content and simply start firing on real drafts. The smoke delve's deterministic subgraph already excludes stuck, locked, and secret doors, so synthesized door states cannot break the delve.

### 5. Treasure grammar extensions — `assemble.py`

Two shapes, both recorded in phase 5's amendment from the owner's B3 spot-check, both deterministic assembly changes with no model interaction:

- **Comma-grouped numbers.** `"1,000 cp"` parses as 1000 — the number lexing under `_COIN` and `_WORTH` accepts comma thousands-separators, and the "no digits outside coin matches" guard counts a comma-grouped number as one match.
- **Quantified each.** `"3 gems worth 50 gp each"` (and the `"each worth 50 gp"` ordering) parses as N valuables of the stated value, kind by the existing gem lexicon — the `each`/`per` early guard narrows to what it was actually protecting against: `per`-anything and *unquantified* `each` (per-monster and conditional treasure) stay unparsed, a dice quantity still trips the dice guard first, and a literal count distributes.
- The grammar stays pinned and conservative — these are the two shapes the phase 5 finding says it "arguably intends to cover," nothing more. Acceptance is measured, not promised: table-driven unit tests pin both shapes plus the guards that must not loosen, the committed corpus's treasure numbers move only through the sweep, and the B3 re-conversion's `treasure_unparsed` count (37 today) lands in the amendment as the before/after.

### 6. Measurement, fixtures, and the standing obligations

- **Fixture re-records.** The survey and content prompt/schema changes strand every recorded fixture that embeds them; the re-record runs through `tools/extract/run_extraction.py` per its README, owner-executed (the phase 5 permission note stands: the classifier refuses agent-side live calls; anything live goes through the owner).
- **The eval regression rule fires in full.** Double corpus sweep, first run committed to `tools/eval/corpus/scoreboard.json` regardless of outcome, second run refreshes the living noise-band table in `tools/eval/README.md`; any out-of-band drop carries its justification in the PR description. The JN1 baseline re-bless from work item 1 rides the same commit discipline.
- **BYOM re-runs.** The four private-corpus modules re-convert (B3 with the recorded `--set 'blank_page_renders=[21]'`) and re-score against unchanged truth files — the same yardstick, the new pipeline. Aggregate before/after numbers (area alignment on B4 under the fixed scorer, `treasure_unparsed` counts, transition and door counts, `area_unreachable` finding counts from `check`) land in the amendment; no retail text anywhere, publication posture unchanged.
- **Consumer acceptance, recorded not gated.** The re-converted drafts load into osr-web's library (its `adventures/` dir already symlinks forge workdirs); the amendment records the structural walkthrough — B3's level 2 entered by stairs, a door opened or forced, DCC #81's lower levels reached — as evidence the milestone means what it says. osr-web changes nothing; that is the point of the contract.
- **Cost envelope:** fixture re-records plus two sweep runs plus four BYOM conversions ≈ $8–10 at phase 5's measured rates; ≤ $15 for the phase.

### 7. Tests

- Schema: `AreaConnection` round-trips with `via`, door conditions, and `to_level`; the at-least-one-target and door-fields-only-on-doors validators reject what they must; survey caches round-trip `description` and `services`.
- Resolution and geometry: level-targeted links become reciprocal transitions with the pinned landing policy (first-in-survey-order, reciprocal-mention pairing, kind derivation, the flag); door edges land on the source-boundary edge for routed corridors and the shared edge for adjacencies; duplicate-mention door facts keep first mention; synthesized output still satisfies osrlib's structural rules (the existing invariant assertions extend to door and transition emission).
- Treasure: table-driven cases for both new shapes, both orderings of `each`, and the guards that must hold (`per`, unquantified `each`, dice quantities, digits outside coin matches).
- Scorer: overlap alignment on B4-shaped synthetic cases (many truth levels into few extracted levels and the reverse, zero-overlap levels, tie-breaks); the committed-corpus equivalence test (number-alignment and overlap-alignment agree on the committed caches); the JN1 pinned baseline re-blessed.
- Report: the new flag serializes, parses, and renders in the generated badge vocabulary; assembly emits it exactly where the landing was guessed.
- All green under the full gate on both OSes, no network anywhere; goldens re-bless only where the plan says they move, with the why in the commit message.

## Sequencing

1. Scorer alignment by overlap, its tests, the equivalence pin, and the JN1 re-bless (work item 1) — the yardstick is fixed before anything is measured through it.
2. Content and survey schema widening with prompt changes (work items 2 and 3), then the owner-run fixture re-records — extraction speaks the new schema before anything consumes it.
3. Geometry doors and transitions, the flag, and assembly's metadata fill (work item 4).
4. Treasure grammar (work item 5) — independent of 2–3, lands here so the BYOM re-run measures everything at once.
5. The measurement pass: double sweep, band refresh, BYOM re-runs and re-scores, the osr-web walkthrough (work item 6); spec, README, and docs impacts; the amendment; the traceability pass.

## Definition of done

- `uv sync && uv run ruff format --check && uv run ruff check && uv run pyright && uv run pytest` passes locally and in CI on both OSes, no network in any test; `mkdocs build --strict` stays green.
- The content schema carries `via`, door conditions, and level-targeted vertical links; geometry synthesizes door edges, secret doors, and guessed-landing transitions flagged `transition_guessed`; assembly fills `Adventure.description` and `TownSpec.services` from the survey; the treasure grammar parses both recorded shapes without loosening its guards — each behavior pinned by the tests named above.
- The scorer aligns levels by area-key overlap, the committed-corpus equivalence test holds, and the JN1 pinned baseline is re-blessed.
- Every recorded fixture stranded by the prompt/schema changes is re-recorded; the double sweep is committed (first run as-is) with the refreshed noise band in `tools/eval/README.md`; any out-of-band metric move carries its justification in the PR description.
- The four BYOM modules are re-converted and re-scored against unchanged truth files, with before/after structure numbers (B4 area alignment, B3 `treasure_unparsed`, per-module door, transition, and `area_unreachable` counts) in the amendment — publication posture unchanged from phase 5.
- The spec's roadmap entry 7, the stage 3/5 rewordings, the flag-vocabulary addition, the open-questions rewrite (custom monsters with the verified osrlib seam gap, party extraction added, vision-first geometry narrowed), and the README/docs impacts are applied with the implementation PR; every item above is traceable to code and tests or to a named amendment entry.
