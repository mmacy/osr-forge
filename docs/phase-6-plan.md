# Phase 6 plan — playable structure

Implementation plan for phase 6 of [the osr-forge spec](spec.md): doors, stairs, and the other structural signal the pipeline reads but cannot yet say. The first consumer's evidence is unambiguous — across every real conversion in osr-web's adventure library (B3, B4, DCC #81, *The Hole in the Oak*), 100% of emitted edges are `open`, so the consumer's entire door subsystem (open, force, listen, pick, secret-door rendering) is dead code against forge output; B3's level 2 (28 areas) has no transitions at all; DCC #81's levels 2–4 are unreachable from the entrance; and `Adventure.description` and `town.services` are empty in every document. The milestone: **a multi-level module converts to a draft whose printed structure actually plays — every level reachable through real stairs, doors that are doors (secret ones included), module metadata populated — the improvement measured where the phase 5 yardsticks reach and recorded as structural before/after counts where they don't, never vibed.** The roadmap entry for this phase is a spec impact applied with the implementation PR, as phase 5's was.

Five facts shape everything:

- **The output format is not the constraint, and neither is the renderer.** osrlib 1.1.0 already expresses everything this phase emits — `Edge`/`DoorSpec` (normal and secret, stuck, locked), `TransitionSpec`, `TownSpec.services`, `Adventure.description` — and the preview renderer has drawn door ticks and transition glyphs since phase 2, built deliberately ahead of synthesis for the override path. No osrlib change is needed anywhere in this phase; the one conversion capability that *does* need an osrlib seam (custom monster emission) is scoped out below with its prerequisite named.
- **The model already reports the missing structure; the schema drops it on the floor.** B3's stage cache carries area 48's vertical connection to `"Second Level"`, and DCC #81's carries `"rock passage"`, `"secret entrance in nearby passage"` — level-shaped and mechanism-shaped answers that `AreaConnection`'s area-keyed `to_key` cannot resolve, so they fall out as `connection_ambiguous` and whole levels orphan. Doors are the same story told in reverse: the spec pinned "door placement is override territory in v1 — the extraction schema carries no door signal," so prose like "a stuck door in the north wall" extracts as a bare direction. Phase 6 is mostly about widening one small schema so the answers the model is already giving have somewhere to land.
- **Phase 5 built the baseline this phase is judged against — and named its customers.** Prompts and schemas have been frozen since phase 1; the phase 5 amendment records the survey mode-flip (1 in 4 known JN1 runs) and the treasure grammar's two failing shapes (B3 carries 37 `treasure_unparsed` flags, named magic items among them) as the two candidates "now judgeable against this BYOM baseline." This phase is that judgment: every prompt, schema, and grammar change here re-runs the sweep under the standing regression rule, and the BYOM four re-convert and re-score against the same truth files.
- **The yardstick has a recorded crack, and it gets fixed before anything is measured through it.** Phase 5's B4 finding: the scorer aligns levels by number, so a legitimate level-modeling disagreement (truth asserts the module's 10 tiers; extraction grouped the same 117 keys into the 6 printed dungeon levels) scored 0.0171 area recall against a draft that had found everything. Survey-discipline changes land in exactly that blast radius, so the alignment fix is this phase's first work item, sequenced before any prompt is touched.
- **The API freeze is owner-waived.** On 2026-07-16 the owner declared both osrlib's frozen-API posture and osr-forge's additive-only artifact-contract fence non-binding while no external consumers exist — the only consumers today are the owner's own projects. The waiver is recorded here so the plan does not silently contradict the repo text it supersedes (spec impacts below carry the rewordings), and it changes framing more than design: every phase 6 change was already the clean shape, and happens to be additive; what the waiver removes is ceremony as a reason — nothing in this plan, and nothing in the phase 7 disposition, may justify a decision by API-compatibility hedging.

## Scope

In scope:

- Connection structure in the content schema — `via` (door, secret door, stairs, trapdoor, chute), door state, and level-targeted vertical links — with the prompt changes to populate it and the consumer changes (geometry bookkeeping, scorer guard) the widened shape requires
- Survey additions — the module's own description, town services — and survey site discipline: the phantom-dungeon rule (B4's five single-area letter-feature "dungeons") and the multi-lair counting rule (the mode-flip)
- Geometry synthesis of door and secret-door edges and of transitions from level-targeted links, with pinned, total placement and landing policies and their report flags
- Assembly: metadata fill, the two treasure-grammar extensions, and the flag-vocabulary addition
- The scorer's level alignment, rebuilt on area-key overlap (the B4 fix), with the JN1 pinned baseline re-blessed
- The full measurement pass the standing rules require: fixture re-records, a double corpus sweep with a refreshed noise band, and BYOM re-runs — all live work owner-executed per the phase 5 permission note

Out of scope, each with its disposition:

- **Custom monster emission** — the fix for DCC #81's measured 0.0 resolution accuracy (every encounter a flagged SRD stand-in) — is the named phase 7 candidate. Its prerequisite is now verified rather than assumed: osrlib has no custom-catalog seam today (`GameSession.new` hardcodes `load_monsters()`/`load_equipment()`, and roughly twenty engine call sites reach for the module-level loaders directly), so bundled `MonsterTemplate`s are an osrlib change first — designed outright under the owner's API waiver, run through osrlib's own two-PR phase loop — and an osr-forge stat-block extraction phase second. With the freeze waived, the deferral stands on scope alone, and squarely: stat-block extraction (AC, HD, attacks, damage, saves, morale, movement, XP from the printed page), template mapping, resolution-metric semantics, and new override kinds are a phase-sized capability in themselves, roughly the size of everything else in this plan combined. The spec's open question is rewritten to carry this finding (spec impact below).
- **Door and transition eval metric families** — deferred. Measuring door recall or transition recall requires truth-format assertions (`via` on truth connections, cross-level link assertions) and re-authoring across five-plus truth files; this phase measures structure through the playability lint (`area_unreachable` counts on multi-level drafts collapse when transitions synthesize) and the connections family — which changes only enough not to break: level-targeted connections are outside its same-level edge universe and are skipped (work item 3), with edge semantics and denominators untouched. The truth-format cost is recorded so a future phase can pick it up deliberately.
- **Vision-first geometry** — unchanged open question. This phase's door and stair signal comes from the key's prose, not the map image; a map-vision pass remains the future candidate for geometry that matches the printed cartography.
- **Party and pregen extraction** — new open question, recorded from consumer evidence (osr-web hardcodes a six-member roster and per-class equipment kits because `adventure.json` has no party surface; osrlib already has `party_to_document`). Not this phase.
- **`travel_turns` extraction** — stays empty. Modules rarely state town-to-dungeon travel in a convertible form, and the first consumer gates town commands on mode, not the field; inventing a number would violate the no-custom-content posture.
- **The survey mode-flip as a solved problem** — this phase changes the prompt and measures; it does not promise the flip's elimination. The targets and their miss disposition are pinned in work item 2.
- **OCR fallback and non-PDF inputs** — spec open questions, unchanged.

Spec impacts, applied with the implementation PR:

- **§ Roadmap** gains entry 7: *Phase 6 — playable structure.* Door and stair extraction and synthesis, survey site discipline, module metadata, treasure-grammar extensions, and overlap-based eval level alignment. Milestone: a multi-level module converts to a draft whose printed structure plays — every level reachable, doors that are doors — measured against the phase 5 baselines where the metrics reach, recorded as structural counts where they don't.
- **§ Pipeline stage 2 (survey)** adds the module description and town services to the identify list; **stage 3 (content)** adds the connection structure to the extracted-facts list; **stage 5 (geometry)** replaces "emit open `Edge`s (door placement is override territory in v1 — the extraction schema carries no door signal; a map-vision pass is the future automatic candidate)" with the door/transition synthesis described here, overrides remaining the last word.

- **§ Open questions**: the custom-monster entry is rewritten to record the verified osrlib seam gap and the two-repo sequencing; a party/pregen extraction entry is added; the vision-first geometry entry is narrowed to map-image geometry (prose-signal doors and stairs are no longer part of the question).
- **§ Non-goals and `AGENTS.md`**: the spec's "osrlib is frozen-API" characterization and `AGENTS.md`'s artifact-contract paragraph each gain the owner's waiver — the fences are suspended while no external consumers exist, the separation rationale (sans-I/O, a converter stays outside the engine) standing on its own. In `AGENTS.md` the waiver *supersedes* the "an installable package *is* external consumption" trigger clause rather than annotating it — the owner's declaration overrides that phase 4 ruling until the owner deems external consumers to exist; appending a no-external-consumers condition beside a clause defining the PyPI release as external consumption would leave the paragraph contradicting itself. ## Work items

### 1. Scorer level alignment by area-key overlap — `evals.py`

- Within an aligned dungeon, truth levels align to extracted levels by **maximal area-key overlap, many-to-one from the truth side**: each truth level independently pairs with the extracted level sharing the most canonical-slug area keys, ties broken by smaller level-number distance and then by lower extracted level number; a truth level with zero overlap everywhere stays unmatched. Several truth levels may pair with one extracted level — that is the B4 shape (10 printed tiers grouped by extraction into the 6 printed coarse levels), and it is why one-to-one matching cannot heal it. Area alignment within each pairing is unchanged, and an extracted area still matches at most one truth area (pairings are processed in truth-level order; a key already claimed is not claimed again). Extracted areas no truth area claims count against precision exactly as today. Dungeon alignment and the phase 5 dungeon-count fields are untouched — B4's disagreement lives entirely at the level tier.
- One hazard, recorded: overlap alignment assumes area keys distinguish levels. A module that keys every level 1..N and extracts partially could overlap a truth level onto the wrong extracted level — a cross-pairing number alignment cannot make. The tie-breaks (number distance first) absorb the equal-overlap cases, the committed corpus is immune (verified: no key recurs across levels within any committed dungeon), and a BYOM module with per-level 1..N keying is a known caveat for its amendment entry, not a blocker.
- The committed corpus scores identically under both rules — not "by construction" but by a verified property of all three members: minimod's single dungeon is single-level; JN1's Abandoned Manor (levels 1–2, keys 77–97/98–106) and JN2's Reed Marsh Temple (levels 1–2, keys 1–8/9–14) are the only multi-level dungeons, and neither has any cross-level key overlap, so maximal overlap picks the equal-numbered level every time. A repo test pins the property forward without keeping the old code: on the committed caches, every matched truth level pairs with the extracted level of equal number (the old rule's outcome, asserted as data — no number-alignment implementation survives). The JN1 pinned baseline therefore re-blesses with unchanged numbers, and the noise band carries until this phase's sweep refreshes it. Sequenced first: the survey-discipline work below moves level groupings by design, and it must be measured through the fixed yardstick, not the cracked one.
- Re-scoring the private BYOM board is offline and free (`score --corpus ~/Documents/osr-forge-byom --workdir …`); B4's truth tiers each sit inside exactly one extracted coarse level with module-wide-unique keys, so its 117 keys align and the 0.0171 artifact resolves into a real measurement. The refreshed numbers land in this phase's amendment. Publication remains deferred on the owner-sampling leg exactly as phase 5 left it — nothing here changes that posture.

### 2. Survey: metadata and site discipline — `survey.py`, `contracts/stages.py`

- **Schema additions, both additive to the survey cache with defaults so every pre-phase-6 workdir still assembles.** `SurveyIndex` gains `description: str = ""` — the module's own pitch, an excerpt of its printed introduction or back-cover text, quoted or tightened from the module's own words, never invented, empty when the module has none. `TownInfo` gains `services: tuple[str, ...] = ()` — the named establishments and services the module states (an inn, a temple, a general store). Assembly maps them onto `Adventure.description` and `TownSpec.services`, both today's hardwired defaults.
- **The phantom-dungeon rule.** B4's survey coined five single-area dungeons from unkeyed letter features on the underground-city map (b/k/o/q/r) — each became a stub "dungeon" with an Enter button in osr-web. The prompt's dungeon definition gains the rule: a dungeon exists only where the module prints a keyed area list for it; lettered or unkeyed map callouts inside another site's map are that site's features, not sites.
- **The multi-lair counting rule.** The mode-flip's shape is known (JN1's ten lairs collapsing into one dungeon, 1 run in 4): the existing one-dungeon-per-keyed-site instruction gets a counting-anchored restatement — enumerate the keyed sites first, then emit exactly that many dungeons. Wording is implementer latitude.
- **Targets, not merge gates — pinned so nobody wonders at sweep time.** The phase's survey targets: JN1 surveys 14/14/14 dungeons in both sweep runs, and a B4 re-conversion surveys one dungeon. A miss does not block the merge and the run is kept — re-rolling an unflattering run is cherry-picking, and the phase 5 dungeon-count fields make any miss legible in the committed record. What a miss triggers is a recorded owner decision in the amendment: accept and record, or iterate the prompt once more (a further fixture re-record and sweep, costed against the envelope). The definition of done binds to targets-hit-or-miss-recorded, not to the targets themselves.
- Both are frozen-prompt changes — the whole point of phase 5's baseline. The eval regression rule fires (work item 6), and every recorded fixture whose request embeds the survey prompt or schema strands and re-records.

### 3. Connection structure in the content schema — `contracts/stages.py`, `content.py`, `evals.py`

`AreaConnection` widens from `(to_key, direction)` to carry mechanism and level-shaped targets — every field an answer the model is already trying to give:

- `via: Literal["passage", "door", "secret_door", "stairs", "trapdoor", "chute", "other"] = "passage"` — the stated mechanism, `passage` when the text names none. `secret_door` is its own value, not a modifier: it drives both `DoorSpec(kind="secret")` and the playability lint's `secret_only_access` warning, which becomes meaningful for the first time. The content prompt's features rule moves with it: secret doors leave the features example list ("altars, pools, levers, secret doors") and become connections when they join keyed areas; a secret door that conceals no keyed connection (a false closet) stays a feature. The prompt edit is owned here, explicitly.
- `door_stuck: bool = False`, `door_locked: bool = False` — stated door conditions.
- `to_key: str | None` and `to_level: int | None` — the escape hatch for exactly the B3 failure: "stairs descend to the second level" states a level, not a keyed area. The prompt instructs: prefer the keyed target; else state the level; the free-string-in-`to_key` fallback stays for anything else and keeps flowing to `connection_ambiguous` as today.
- **Failure posture: tolerate and flag, never reject — the house rule for schema-legal junk.** The batch JSON schema stays flat (no conditional coupling; structured-output implementations handle it badly), so a schema-valid response can carry door conditions on a non-door `via`, or neither target. The cache models accept all of it; consumers discard-with-flag, mirroring the `count_dice` and empty-monster-name precedents: geometry reads door conditions only when `via` is a door kind (stray conditions are ignored), and a connection with neither `to_key` nor `to_level` is skipped with `connection_ambiguous:no target stated` on its area. No cache validator rejects what the schema admits — a pydantic error mid-stage is a crash, not defense.
- **The consumers of the widened shape change with it, named here so nothing crashes by omission.** Geometry's unresolved bookkeeping formats its flag detail from `to_key` only when one was given (a `to_level`-only link that cannot be realized gets its own disposition, work item 4). The scorer's connections family guards the same null: a connection with `to_key=None` is outside the same-level edge universe and is skipped before `canonical_slug` ever sees it — without this, the phase's own sweep crashes on the first cache the new prompt produces. Edge semantics and denominators are otherwise untouched.
- **Duplicate-mention merging gets the direction rule's upgrade clause, not a blunter one.** The default `via="passage"` is an absence, not an assertion — a later mention stating a mechanism fills it, exactly as a later stated compass direction fills an earlier `None` today. The first *stated* mechanism wins over subsequent conflicting statements, and door conditions merge the same way (first stated). *(Amended during implementation, where the plan was silent: the clause applies to keyed cross-level links too — dedup keeps the first mention's endpoints and direction while a later mention's stated mechanism fills an earlier absence — and a one-way mechanism (trapdoor, chute) filling the absence re-orients the link to its stating mention's end: a chute drops from where the page says it drops, and realizing it from the passage-stating end would emit it from the wrong room.)*
- Prompt guidance names the signals: doors, secret doors, stairs, trapdoors, chutes, stuck/locked conditions, and the level-vs-area target preference. Same-level resolution, batching, and the direction rules are untouched.

### 4. Geometry: doors and transitions — `geometry.py`, `assemble.py`

- **Door edges — a total rule over the three ways a connection materializes.** A resolved same-level connection either (a) realizes as the route placement creates for a tree edge — the door is the first edge the route opens leaving the source area's cell cluster, whatever the corridor's length, including length zero when the rooms sit flush (the prose describes the door in the wall of the room being described, so the source end is the faithful placement); or (b) realizes with the roles swapped when the router re-anchored the child onto an earlier room — the stated door then has no edge on the described wall, and the door fact is dropped with `connection_ambiguous:door to <key> not placed` on the source area; or (c) does not materialize at all — a connection that closes a cycle places no route today — and the door fact is dropped with the same flag. In cases (b) and (c) the geometry `edges` override is the designed remedy, and the flag is what tells the human to reach for it. Where the rule fires (case a), the opened edge becomes `Edge(kind="door", door=DoorSpec(kind=…, stuck=…, locked=…))`.
- **Transitions from level-targeted links — a total landing policy.** A vertical connection with `to_level` and no resolvable `to_key` first validates its target: a level the dungeon doesn't have, the source's own level, or a level with zero surveyed areas drops the link with `connection_ambiguous:level <N>` on the source area. Surviving links between the same level pair merge pairwise in survey order when they point in opposite directions — each merged pair yields one reciprocal transition between the two mentioning areas (both ends were stated as stair-bearing; the guess is only that they are the same stairway). Leftover links land on the target level's first keyed area in survey order. Every `to_level`-derived transition flags its source area `transition_guessed:<dungeon>/<level>/<area>` (the chosen far end) — merged pairs flag both mentioning areas — and the existing geometry `transitions` override corrects any landing. The vertical sense: a stated `up`/`down` direction wins; otherwise the level numbers decide (a higher target number is down — deeper levels number higher throughout the pipeline). The kind derives from `via` (`stairs` → `stairs_up`/`stairs_down` by that sense, `trapdoor` and `chute` as themselves); a vertical link whose `via` is anything else defaults to stairs, the overwhelmingly common printed mechanism.
- **Reciprocals follow osrlib's own one-way semantics.** Stairs emit the reciprocal return transition as the keyed-target path does today; `trapdoor` and `chute` emit none — they are one-way by design, which is exactly why the lint already exempts them from `transition_unpaired`. (The previous behavior — synthesizing return stairs unconditionally — would manufacture stairs back up a chute.)
- **The flag.** `transition_guessed` is a new `Flag` member — an addition to `report.json`'s vocabulary (additive anyway, waiver aside), the thing a review UI badges so a human confirms the landing or corrects it. The spec's flag-vocabulary sentence, the hand-written description dict in `tools/docs/gen_vocabulary.py`, and the generated badge table update with it.
- **No preview or lint work.** The SVG renderer has drawn door ticks (outlined when secret) and transition glyphs since phase 2; `secret_only_access` and `transition_unpaired` were specified for exactly this content and simply start firing on real drafts. The smoke delve's deterministic subgraph already excludes stuck, locked, and secret doors, so synthesized door states cannot break the delve.

### 5. Treasure grammar extensions — `assemble.py`

Two shapes, both recorded in phase 5's amendment from the owner's B3 spot-check, both deterministic assembly changes with no model interaction:

- **Comma-grouped numbers.** `"1,000 cp"` parses as 1000 — the number lexing under `_COIN` and `_WORTH` accepts comma thousands-separators, and the "no digits outside coin matches" guard counts a comma-grouped number as one match.
- **Quantified each.** `"3 gems worth 50 gp each"` (and the `"each worth 50 gp"` ordering) parses as N valuables of the stated value, kind by the existing gem lexicon — the `each`/`per` early guard narrows to what it was actually protecting against: `per`-anything and *unquantified* `each` (per-monster and conditional treasure) stay unparsed, a dice quantity still trips the dice guard first, and a literal count distributes.
- The grammar stays pinned and conservative — these are the two shapes the phase 5 finding says it "arguably intends to cover," nothing more. Acceptance is measured, not promised: table-driven unit tests pin both shapes plus the guards that must not loosen, the committed corpus's treasure numbers move only through the sweep, and the B3 re-conversion's `treasure_unparsed` count (37 today) lands in the amendment as the before/after.

### 6. Measurement, fixtures, and the standing obligations

- **Fixture re-records.** The survey and content prompt/schema changes strand every recorded fixture that embeds them; the re-record runs through `tools/extract/run_extraction.py` per its README, owner-executed (the phase 5 permission note stands: the classifier refuses agent-side live calls; anything live goes through the owner).
- **The eval regression rule fires in full.** Double corpus sweep, first run committed to `tools/eval/corpus/scoreboard.json` regardless of outcome, second run refreshes the living noise-band table in `tools/eval/README.md`; any out-of-band drop carries its justification in the PR description. The JN1 baseline re-bless from work item 1 rides the same commit discipline.
- **Changelog discipline.** Nearly everything in this phase is user-visible behavior; the implementation PR adds its bullets to `CHANGELOG.md`'s `[Unreleased]` section, per the third standing obligation.
- **BYOM re-runs.** The four private-corpus modules re-convert (B3 with the recorded `--set 'blank_page_renders=[21]'`) and re-score against unchanged truth files — the same yardstick, the new pipeline. Aggregate before/after numbers (area alignment on B4 under the fixed scorer, `treasure_unparsed` counts, transition and door counts, `area_unreachable` finding counts from `check`) land in the amendment; no retail text anywhere, publication posture unchanged.
- **Consumer acceptance, recorded not gated.** The re-converted drafts load into osr-web's library (its `adventures/` dir already symlinks forge workdirs); the amendment records the structural walkthrough — B3's level 2 entered by stairs, a door opened or forced, DCC #81's lower levels reached — as evidence the milestone means what it says. osr-web changes nothing; that is the point of the contract.
- **Cost envelope:** fixture re-records plus two sweep runs plus four BYOM conversions ≈ $8–10 at phase 5's measured rates, with headroom for one owner-elected survey-prompt iteration (work item 2's miss disposition); ≤ $15 for the phase.

### 7. Tests

- Schema: `AreaConnection` round-trips with `via`, door conditions, and `to_level`; the tolerate-and-flag posture is pinned (door conditions on a non-door `via` load and are ignored by geometry; a connection with neither target loads and assembly flags `connection_ambiguous:no target stated`); survey caches round-trip `description` and `services`, and a pre-phase-6 survey cache without them still loads and assembles.
- Resolution and geometry: level-targeted links validate, merge pairwise, land, and flag per the pinned policy (invalid levels, self-links, empty levels, opposite-direction pairing in survey order, first-area fallback, both-ends flagging on merged pairs, vertical sense from stated direction then level numbers); trapdoor and chute links emit no reciprocal while stairs do; door edges land on the first opened route edge out of the source cluster for tree edges including flush placements, and the re-anchor and cycle cases drop the door fact with the pinned flag; duplicate-mention merging upgrades a default `passage` and keeps the first stated mechanism and conditions; synthesized output still satisfies osrlib's structural rules (the existing invariant assertions extend to door and transition emission).
- Treasure: table-driven cases for both new shapes, both orderings of `each`, and the guards that must hold (`per`, unquantified `each`, dice quantities, digits outside coin matches).
- Scorer: overlap alignment on B4-shaped synthetic cases (many truth levels onto one extracted level, the reverse split, zero-overlap levels, claimed-key exclusivity, tie-breaks); the committed-corpus property test (every matched truth level pairs with the equal-numbered extracted level on the committed caches — the old outcome asserted as data, no second implementation kept); connections scoring skips `to_key=None` without touching edge denominators; the JN1 pinned baseline re-blessed.
- Report: the new flag serializes, parses, renders in the generated badge vocabulary, and carries its description in `gen_vocabulary.py`; assembly emits it exactly where a landing was guessed.
- All green under the full gate on both OSes, no network anywhere; goldens re-bless only where the plan says they move, with the why in the commit message.

## Sequencing

1. Scorer alignment by overlap, its tests, the committed-corpus property pin, and the JN1 re-bless (work item 1) — the yardstick is fixed before anything is measured through it.
2. Content and survey schema widening with prompt changes and the named consumer guards (work items 2 and 3), then the owner-run fixture re-records — extraction speaks the new schema before anything consumes it, and every consumer of the widened shape changes in the same commit that widens it.
3. Geometry doors and transitions, the flag, and assembly's metadata fill (work item 4).
4. Treasure grammar (work item 5) — independent of 2–3, lands here so the BYOM re-run measures everything at once.
5. The measurement pass: double sweep, band refresh, BYOM re-runs and re-scores, the osr-web walkthrough (work item 6); spec, README, and docs impacts; the amendment; the traceability pass.

## Definition of done

- `uv sync && uv run ruff format --check && uv run ruff check && uv run pyright && uv run pytest` passes locally and in CI on both OSes, no network in any test; `mkdocs build --strict` stays green.
- The content schema carries `via`, door conditions, and level-targeted vertical links under the tolerate-and-flag posture; geometry synthesizes door edges, secret doors, and guessed-landing transitions under the total policies of work item 4; assembly fills `Adventure.description` and `TownSpec.services` from the survey; the treasure grammar parses both recorded shapes without loosening its guards — each behavior pinned by the tests named above.
- The scorer aligns levels by area-key overlap (many-to-one from the truth side), skips level-targeted connections in the connections family, holds the committed-corpus property test, and the JN1 pinned baseline is re-blessed.
- Every recorded fixture stranded by the prompt/schema changes is re-recorded; the double sweep is committed (first run as-is) with the refreshed noise band in `tools/eval/README.md`; any out-of-band metric move carries its justification in the PR description; the changelog bullets land in the same PR.
- The survey targets (JN1 14/14/14 in both runs; B4 surveys one dungeon) are hit, or each miss is recorded in the amendment with the kept run and the owner's disposition (accept, or one costed prompt iteration).
- The four BYOM modules are re-converted and re-scored against unchanged truth files, with before/after structure numbers (B4 area alignment, B3 `treasure_unparsed`, per-module door, transition, and `area_unreachable` counts) in the amendment — publication posture unchanged from phase 5.
- The spec's roadmap entry 7, the stage 2/3/5 rewordings, the flag-vocabulary addition, the open-questions rewrite (custom monsters with the verified osrlib seam gap, party extraction added, vision-first geometry narrowed), the owner-waiver rewordings in the spec's non-goals and `AGENTS.md`, and the README/docs impacts are applied with the implementation PR; every item above is traceable to code and tests or to a named amendment entry.
