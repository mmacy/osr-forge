# Test assets

Every asset in this directory documents its provenance and license here. Test
assets live outside the built distribution ‚Äî the osr-forge wheel ships no game
content.

## minimod/

An original mini-module, *The Root Cellar of Old Wenna*, authored for this
repository as a test asset and dedicated to the public domain under
[CC0 1.0](https://creativecommons.org/publicdomain/zero/1.0/).

- `minimod.pdf` ‚Äî 5 pages. Pages 1‚Äì3 and 5 carry a text layer (keyed areas,
  stated room dimensions, monster names); page 4 is a raster map image with no
  text layer, exercising the scanned-module path.
- `encrypted.pdf` ‚Äî a 1-page password-protected PDF exercising the
  encrypted-source error path. The password is `minimod-owner-password` (it is
  never used; the tests only assert that opening without it fails).
- `pages/` ‚Äî the exact page renders and text layers the extraction fixtures
  were recorded against (phase 1's recording session).
- `fixtures/` ‚Äî recorded `survey` and `content` fixtures from a real
  `preprocess ‚Üí survey ‚Üí content` run over `minimod.pdf`. There is no
  `monsters` fixture and none is needed: minimod's whole encounter-name
  population resolves in the exact tier, so the monsters stage makes no model
  call (the pipeline-replay test would fail loudly if it ever did).
- `expected/` ‚Äî the full-chain goldens the pipeline-replay test pins
  byte-for-byte: the stage caches (`survey.json`,
  `areas.<dungeon>.<level>.json`, `monsters.json`) at the top level, plus
  `adventure.json`, `report.json`, and `previews/`. The re-bless rule: an
  osrlib or osr-forge version bump changes `adventure.json`'s stamped envelope
  and `report.json`'s `osrforge_version` ‚Äî regenerate deliberately by
  re-running the fixture chain (the pipeline-replay test's `run_pipeline`
  produces exactly these files), never hand-edit.

The PDFs are generated by `tools/minimod/generate_minimod.py` (reportlab +
Pillow). Regenerate from the repo root with:

```sh
uv run tools/minimod/generate_minimod.py
```

Regenerating changes the PDF bytes (creation timestamps); nothing byte-compares
the committed PDFs, so that's harmless ‚Äî but don't regenerate without a reason.

**Fixture coupling:** regenerating `minimod.pdf`, bumping pypdfium2 or Pillow,
or editing an extraction prompt or schema strands `pages/`, `fixtures/`, and
`expected/` together ‚Äî request fingerprints hash the page bytes and the prompt
text. Re-record the whole set via the minimod session commands in
`tools/extract/README.md`; never rebuild fixture requests from fresh renders.

## chaotic-caves/

*JN1 The Chaotic Caves* ‚Äî the real-module evidence set; see
`chaotic-caves/README.md` for provenance, license, and the committed layout.
Two entries specific to the correction loop (phase 3):

- `overrides.yaml` ‚Äî the phase 3 correction session's file, and itself a test
  asset: the milestone gate assembles the committed caches plus this file and
  byte-compares against `expected-corrected/`. **Editing it re-blesses the
  corrected goldens** (regenerate via the documented command below). CC BY-SA
  text derived from the module rides the directory's existing license fence.
- `expected-corrected/` ‚Äî the corrected goldens: `adventure.json` and
  `previews/` as `assemble` writes them over caches + overrides, and
  `report.json` as `check` rewrites it (the post-check report, findings
  merged ‚Äî the session's accepted warnings are byte-pinned here).

The corrected-goldens re-bless rule: after any deliberate change to assembly,
overrides semantics, the lint, or osrlib, regenerate ‚Äî never hand-edit:

```sh
uv run tools/extract/run_extraction.py goldens \
    --stages-dir tests/assets/chaotic-caves/stages \
    --out tests/assets/chaotic-caves/expected-corrected \
    --page-count 48 \
    --overrides tests/assets/chaotic-caves/overrides.yaml
```

The uncorrected `expected/` goldens regenerate with the same command minus
`--overrides` (their `report.json` is the post-assemble report; the runner
handles the difference).
